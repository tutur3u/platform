The short version is:

- `apps/web/src/proxy.ts` is the current edge/router coordinator for workspace-aware URL handling.
- `personal` and `internal` are canonical browser slugs; database queries should still use UUID workspace IDs.
- Workspace `handle` (slug) can be used as a join-discovery input (for example onboarding and workspace selector), and server-side workspace lookup helpers must accept valid handle-format slugs (`^[a-z0-9](?:[a-z0-9_-]{0,62}[a-z0-9])?$`) in addition to UUID, `personal`, and `internal`.
- `WorkspaceWrapper` is the server-page entry point for workspace resolution.
- `normalizeWorkspaceId()` is the route-handler entry point for workspace-scoped params.
- `getPermissions()` is the fine-grained authorization helper layered on top of routing and membership.
- Root redirects can be user-configured per workspace via `user_workspace_configs` (config id: `ROOT_DEFAULT_NAVIGATION`), but those defaults only apply when the user visits the app root (`/` or a locale root such as `/vi`). Direct workspace home routes such as `/personal` or `/{workspace}` must stay on the workspace home surface instead of auto-opening a configured app or task board. Redirect targets should be validated at request time and stale deep links (for example deleted task boards) must gracefully fall back to a broader route.

Prefer the two architecture pages above for current implementation
details. Note also that `apps/web` (Next.js, port `7803`) is being migrated to
`apps/tanstack-web` (TanStack Start) plus `apps/backend` (Rust); see the
[TanStack + Rust migration](/platform/architecture/tanstack-rust-migration) for
how routing responsibilities are shifting.
