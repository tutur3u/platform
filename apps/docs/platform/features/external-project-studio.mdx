## TL;DR

The EPM surfaces now follow two distinct workflows:

- The **root workspace** owns the canonical registry and workspace bindings through a searchable control tower.
- Each **bound workspace** uses **EPM (External Project Management)** as a two-mode tool:
  - `Preview` mode is the default and renders the bound project from the preview delivery payload.
  - `Edit` mode is the operational control plane for entries, workflow, and settings.

## Root Control Tower

- Use the registry search and adapter filter to find canonical projects quickly.
- Keep project edits in the registry cards so adapter, delivery profile, and recommended collections stay visible while editing.
- Use the sticky action panel on the right for two jobs:
  - create a new canonical project
  - bind or unbind a destination workspace
- Recent binding audits stay below the registry so operational changes remain visible after edits.

## Workspace EPM

- Use a top-level `Preview / Edit` mode switch instead of a tab-heavy admin shell.
- Keep `Preview` as the default landing mode.
  - Render collections and entries from `GET /api/v1/workspaces/:wsId/external-projects/delivery?preview=true`.
  - Keep admin chrome light: mode switch, compact collection controls, and a hidden command menu for secondary actions.
  - Use the full available width for a gallery-style entry surface instead of splitting the page into a rail plus a second inline preview column.
  - Keep gallery cards compact and image-first so more entries fit on screen without turning the index into a dense table.
  - Clicking an entry should open the fullscreen entry editor dialog instead of creating another preview state on the index.
  - Do not show workflow queues, metadata accordions, or settings cards in Preview mode.
- Keep `Edit` as the operational index.
  - Use dropdowns for section, collection, and entry/workflow context so the index stays compact.
  - Keep quick collection and entry creation/deletion in compact menus or inline tiles instead of persistent top-bar buttons.
  - `Entries` is for collection-scoped browsing, search, and opening the fullscreen entry editor dialog.
  - `Workflow` is for draft/scheduled/published/archived queues plus bulk actions.
  - `Settings` is for binding context, import controls, and routing into dedicated collection pages.
- Do not duplicate the same record across featured cards, selected-entry cards, and queue summaries on the index page.
- Entry authoring now uses the fullscreen dialog editor from the main `/epm` surface.
  - Use the dialog for focused title/slug/status edits, schedule changes, cover uploads, and asset ordering.
  - Keep the dashboard as the operator index; do not duplicate the same heavy editor inline across cards or summary panels.
  - Preserve `/{wsId}/epm/entries/{entryId}` as a deep-link or fallback route only when a standalone page is still required.
- Collection details now live at `/{wsId}/epm/collections/{collectionId}`.
  - Use this page for collection title/description/enabled-state changes and quick routing into collection entries.
  - Keep collection editing off the main `/epm` body.
- Cover images are first-class.
  - Use image assets with `sort_order = 0` as the lead cover for preview and admin surfaces.
  - If a lead image cannot resolve from Drive/storage, drop the broken media source and render the non-image fallback treatment instead of leaving broken alt text in the UI.
  - Reordering image assets on the entry detail page changes which image leads the entry across the EPM surfaces.
- Treat the `Workflow` section in Edit mode as the operator queue.
  - Multi-select lives there, not across Preview mode or entry summaries.
  - Schedule, publish, draft, and archive actions should be available without reopening each record.
- Import, create, duplicate, save, publish, and unpublish actions stay explicit. There is no autosave.

## Implementation Notes

- The preview source of truth is the existing preview delivery API, not a duplicate client-side payload builder.
- The workspace studio still uses the existing internal API helpers for entry, block, asset, import, publish, and binding operations.
- The `/epm`, entry-detail, and collection-detail routes should stay as thin server auth gates and fetch their working data from a shared `epm-studio` TanStack Query cache on the client.
- Mutations on the dedicated detail pages should patch or invalidate that shared studio query instead of calling `router.refresh()`.
- When the main `/epm` shell opens the fullscreen entry editor dialog, keep it on the same shared studio query cache so open/close transitions feel instant and do not need route refreshes.
- Cover uploads and replacements reuse the existing workspace asset signed-upload helper; no separate cover-specific backend endpoint is needed.
- The workspace route is now `/{wsId}/epm`, with `/{wsId}/external-projects` redirecting to it for backward compatibility.
- The main `/epm` page is now an index plus mode shell, not a hybrid inline editor.

## Team Guidance

- When changing external-project authoring UX, keep the live preview wired to the preview delivery endpoint instead of inventing a second payload contract in the client.
- When the tool needs to behave like both a front-end preview and an admin workspace, split it into explicit `Preview` and `Edit` modes rather than blending both into the same crowded surface.
- For dense admin surfaces, prefer one compact index plus a near-fullscreen dialog or dedicated page for item-level authoring instead of rebuilding mini-editors inside the index view.
- Keep the detail routes on the same shared studio query cache as the index so save/publish/cover changes stay client-fast without route refreshes.
- For external-project entries that need strong visual identity, treat cover image upload/replacement/reordering as a first-class workflow instead of burying it under generic asset editing.
- Add any new user-facing copy to both `apps/web/messages/en.json` and `apps/web/messages/vi.json`, then run `bun i18n:sort`.
