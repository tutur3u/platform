## TL;DR

The EPM surfaces now follow two distinct workflows:

- The **root workspace** owns the canonical registry and workspace bindings through a searchable control tower.
- Each **bound workspace** uses **EPM (External Project Management)** as a view-first operator dashboard with tabs, compact queues, dedicated entry detail pages, first-class cover management, and on-demand preview.

## Root Control Tower

- Use the registry search and adapter filter to find canonical projects quickly.
- Keep project edits in the registry cards so adapter, delivery profile, and recommended collections stay visible while editing.
- Use the sticky action panel on the right for two jobs:
  - create a new canonical project
  - bind or unbind a destination workspace
- Recent binding audits stay below the registry so operational changes remain visible after edits.

## Workspace EPM

- Use tabs to separate `Overview`, `Content`, `Workflow`, `Activity`, and `Settings`.
- The workspace shell now follows a flatter operator dashboard:
  - the index page is a full-width control surface for summary, routing, and queue management
  - the featured card uses the current entry cover when it exists, but the dashboard no longer tries to be the full editor
  - collection and workflow health panels stay visible without overwhelming the viewport
- Keep the default page view-first.
  - Summary cards should be readable without opening an editor.
  - Metadata and profile payloads should stay collapsed until requested.
  - Editing should open through explicit icon or button affordances that route into a dedicated entry details page.
- Entry details now live at `/{wsId}/epm/entries/{entryId}`.
  - Use this page for focused title/slug/status edits, schedule changes, cover uploads, and asset ordering.
  - Keep the dashboard as the operator index; do not bring the heavy entry editor back inline.
- Cover images are first-class.
  - Use image assets with `sort_order = 0` as the lead cover for dashboard and preview surfaces.
  - If a lead image cannot resolve from Drive/storage, drop the broken media source and render the non-image fallback treatment instead of leaving broken alt text in the UI.
  - Reordering image assets on the detail page should change which image leads the entry across the EPM surfaces.
- Dashboard presentation is locally customizable per workspace.
  - `Dashboard mode` switches between operator-first, visual-first, and workflow-first emphasis.
  - `Density` toggles between compact and comfortable spacing.
  - `Show visual cards`, `Show collection health`, and `Show activity feed` persist in local dashboard preferences.
- Treat the `Workflow` tab as the operator queue.
  - Multi-select lives there, not across every content surface.
  - Schedule, publish, draft, and archive actions should be available without reopening each record.
- Live preview is still on demand.
  - Open it from the sticky action bar when you need a rendered or payload check.
  - **Rendered** should stay lightweight and selection-aware.
  - **Payload** should keep reading from `GET /api/v1/workspaces/:wsId/external-projects/delivery?preview=true`.
- Import, create, duplicate, save, publish, and unpublish actions stay explicit. There is no autosave.

## Implementation Notes

- The preview source of truth is the existing preview delivery API, not a duplicate client-side payload builder.
- The workspace studio still uses the existing internal API helpers for entry, block, asset, import, publish, and binding operations.
- Cover uploads and replacements reuse the existing workspace asset signed-upload helper; no separate cover-specific backend endpoint is needed.
- The workspace route is now `/{wsId}/epm`, with `/{wsId}/external-projects` redirecting to it for backward compatibility.
- Preview still opens in a sheet so the default workspace stays focused on navigation and workflow first.

## Team Guidance

- When changing external-project authoring UX, keep the live preview wired to the preview delivery endpoint instead of inventing a second payload contract in the client.
- For dense admin surfaces, keep collection-level controls inline but move item-level authoring into a dedicated editor page or near-fullscreen dialog. Do not render the same heavy editor twice in both the page body and an overlay.
- In split-view admin pages, prefer one compact selected-item summary in the main column and keep preview as an explicit secondary action when it would otherwise crowd the default workspace.
- For external-project entries that need strong visual identity, treat cover image upload/replacement/reordering as a first-class workflow instead of burying it under generic asset editing.
- Use tooltips for secondary control explanations so the default layout stays compact, especially in sticky action bars and collection controls.
- Add any new user-facing copy to both `apps/web/messages/en.json` and `apps/web/messages/vi.json`, then run `bun i18n:sort`.
