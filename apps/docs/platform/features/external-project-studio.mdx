## TL;DR

The EPM surfaces now follow two distinct workflows:

- The **root workspace** owns the canonical registry and workspace bindings through a searchable control tower.
- Each **bound workspace** uses **EPM (External Project Management)** as a two-mode tool:
  - `Preview` mode is the default and renders the bound project from the preview delivery payload.
  - `Edit` mode is the operational control plane for entries, workflow, and settings.

## Root Control Tower

- Use the registry search and adapter filter to find canonical projects quickly.
- Keep project edits in the registry cards so adapter, delivery profile, and recommended collections stay visible while editing.
- Use the sticky action panel on the right for two jobs:
  - create a new canonical project
  - bind or unbind a destination workspace
- Recent binding audits stay below the registry so operational changes remain visible after edits.

## Workspace EPM

- Use a top-level `Preview / Edit` mode switch instead of a tab-heavy admin shell.
- Keep `Preview` as the default landing mode.
  - Render collections and entries from `GET /api/v1/workspaces/:wsId/external-projects/delivery?preview=true`.
  - Keep admin chrome light: mode switch, compact collection controls, and a hidden command menu for secondary actions.
  - Use the full available width for a gallery-style entry surface instead of splitting the page into a rail plus a second inline preview column.
  - Keep gallery cards compact and image-first so more entries fit on screen without turning the index into a dense table.
  - Clicking an entry should open the fullscreen entry editor dialog instead of creating another preview state on the index.
  - Do not show workflow queues, metadata accordions, or settings cards in Preview mode.
- Keep `Edit` as the operational index.
  - Use dropdowns for section, collection, and entry/workflow context so the index stays compact.
  - Keep quick collection and entry creation/deletion in compact menus or inline tiles instead of persistent top-bar buttons.
  - `Entries` is for collection-scoped browsing, search, and opening the fullscreen entry editor dialog.
  - `Workflow` is for draft/scheduled/published/archived queues plus bulk actions.
  - `Settings` is for binding context, import controls, and routing into dedicated collection pages.
- Do not duplicate the same record across featured cards, selected-entry cards, and queue summaries on the index page.
- Entry authoring now uses the fullscreen dialog editor from the main `/epm` surface.
  - Use the dialog for focused title/slug/status edits, schedule changes, cover uploads, and asset ordering.
  - Keep the dashboard as the operator index; do not duplicate the same heavy editor inline across cards or summary panels.
  - Preserve `/{wsId}/epm/entries/{entryId}` as a deep-link or fallback route only when a standalone page is still required.
- Collection details now live at `/{wsId}/epm/collections/{collectionId}`.
  - Use this page for collection title/description/enabled-state changes and quick routing into collection entries.
  - Keep collection editing off the main `/epm` body.
- Cover images are first-class.
  - Use image assets with `sort_order = 0` as the lead cover for preview and admin surfaces.
  - If a lead image cannot resolve from Drive/storage, drop the broken media source and render the non-image fallback treatment instead of leaving broken alt text in the UI.
  - Reordering image assets on the entry detail page changes which image leads the entry across the EPM surfaces.
  - Use the asset gallery for fast per-image operations: set-as-cover, quick single-image delete, and inline caption editing.
  - Treat per-image captions as asset metadata so external apps can consume the same text without scraping the admin UI.
  - Deleting an entry or an entire collection should also delete any linked workspace Drive objects referenced by those assets' `storage_path` values so EPM does not leave orphaned media behind.
- Treat the `Workflow` section in Edit mode as the operator queue.
  - Multi-select lives there, not across Preview mode or entry summaries.
  - Schedule, publish, draft, and archive actions should be available without reopening each record.
- Import, create, duplicate, save, publish, and unpublish actions stay explicit. There is no autosave.

## Implementation Notes

- The preview source of truth is the existing preview delivery API, not a duplicate client-side payload builder.
- The workspace studio still uses the existing internal API helpers for entry, block, asset, import, publish, and binding operations.
- Collection routing labels for external apps should live in `collection.config.navigation.title` so EPM can manage nav titles without overloading the operator-facing collection title.
- The `/epm`, entry-detail, and collection-detail routes should stay as thin server auth gates and fetch their working data from a shared `epm-studio` TanStack Query cache on the client.
- Mutations on the dedicated detail pages should patch or invalidate that shared studio query instead of calling `router.refresh()`.
- When the main `/epm` shell opens the fullscreen entry editor dialog, keep it on the same shared studio query cache so open/close transitions feel instant and do not need route refreshes.
- After entry asset uploads, deletes, or cover reordering, keep the optimistic cache update for responsiveness but also invalidate the shared `epm-studio` query so the page re-fetches the canonical backend asset rows right away.
- When uploads or media mutations are running in the entry dialog, surface a visible processing indicator near the command bar and media controls instead of leaving the user with silent disabled buttons.
- Cover uploads and replacements reuse the existing workspace asset signed-upload helper; no separate cover-specific backend endpoint is needed.
- Optimize image uploads on the client before requesting the signed upload PUT so gallery and cover assets stay lighter without requiring a separate media-processing job.
- For EPM image surfaces, keep `asset_url` as the original asset route and use `preview_url` for a transformed Supabase variant (for example constrained width/height plus quality). The gallery and editor should prefer `preview_url` first so the workspace ships smaller images by default.
- The SDK should expose the same navigation/media contract external projects need: absolute studio asset URLs, collection navigation-title helpers, asset delete methods, and media metadata fields such as captions.
- For external delivery apps that need client-side navigation or infinite lists, prefetch and hydrate one delivery query near the root layout, then derive route-level view models from that shared cache instead of repeating route-level SDK fetches.
- For featured rails or hero carousels in external delivery apps, derive the ordered items from existing delivery content plus section/profile configuration (for example featured slugs in `profile_data`) rather than inventing a second featured payload contract just for the frontend.
- If no featured slug config is present for a given surface, hide the special featured carousel/rail entirely instead of falling back to the first published entry or image. Keep the normal gallery/list visible as the default experience.
- For Yoola-style external sites, keep sitewide public copy in the `singleton-sections` collection and treat those entries as page configuration:
  - use slugs like `gallery`, `writing`, and `about`
  - use the entry title/subtitle/description for hero framing
  - use the first markdown block for longform page copy
  - keep page-specific structured options such as `socialLinks`, `markers`, `stats`, `navLabel`, or callout panels in `profile_data`
  - manage entry taxonomy from the structured EPM form too: use `profile_data.category` for gallery-style categorical grouping and `profile_data.tags` for ordered tag chips instead of hand-editing those values in raw JSON
  - treat singleton section entries such as `gallery` and `writing` as the canonical taxonomy registry too: persist picker/filter suggestions in `profile_data.categoryOptions` and `profile_data.tagOptions` on that section entry instead of inferring “existing” values from every published entry
  - keep taxonomy creation lightweight: the main control should be the existing-value picker, with a compact adjacent `+` action that reveals inline create inputs only when needed
  - manage featured ordering from the section entry detail form and persist canonical slug arrays in `profile_data` (`featuredArtworkSlugs` for `gallery`, `featuredEntrySlugs` for `writing`) instead of hand-editing raw JSON
  - when a section editor manages featured ordering, keep selection and ordering in one compact list instead of splitting the UI into a "selected" rail plus a second "available" checklist
  - surface direct add/remove featured controls on candidate artwork and writing entry dialogs too, but keep those controls mutating the same singleton section entry so there is still one ordering source of truth
- External apps like Yoola should consume EPM through one server-side adapter module that fetches delivery once, maps it into the app's view model, and passes typed props down to client components. Do not scatter SDK calls or hardcoded mirror content arrays across multiple pages.
- If an external app adopts TanStack Query for delivery, keep that adapter as the normalization layer and hydrate it once at the app shell boundary. Route components should read from the shared query cache and only own incremental rendering state such as infinite scroll windows.
- For Yoola-style writing routes, use the delivery entry's markdown blocks as the canonical longform body and render them directly on the external site instead of maintaining a second static prose source.
- In the EPM entry editor, present the persisted `summary` field as a markdown-capable description/excerpt surface instead of a plain textarea label. Keep longform writing and singleton-page bodies on markdown blocks so external apps can render both short and long copy directly from delivery.
- When publishing `packages/sdk` for external apps, keep the package `types` and `bun` export targets on built `dist/*` declarations/runtime files rather than raw `src/*` source entrypoints. External Next/Bun apps can otherwise resolve internal source files and inherit monorepo-only type assumptions.
- When EPM surfaces render internal asset API URLs such as `/api/v1/workspaces/:wsId/external-projects/assets/:assetId`, bypass the Next.js image optimizer (`unoptimized`) instead of routing them through `/_next/image`. Protected or signed internal asset routes can return optimizer-side `400` responses even though the direct asset URL is valid.
- When bulk-managing files in the workspace Drive explorer, keep selection state stable across search/view changes and expose destructive bulk actions behind explicit confirmation dialogs.
- The workspace route is now `/{wsId}/epm`, with `/{wsId}/external-projects` redirecting to it for backward compatibility.
- The main `/epm` page is now an index plus mode shell, not a hybrid inline editor.

## Team Guidance

- When changing external-project authoring UX, keep the live preview wired to the preview delivery endpoint instead of inventing a second payload contract in the client.
- When the tool needs to behave like both a front-end preview and an admin workspace, split it into explicit `Preview` and `Edit` modes rather than blending both into the same crowded surface.
- For dense admin surfaces, prefer one compact index plus a near-fullscreen dialog or dedicated page for item-level authoring instead of rebuilding mini-editors inside the index view.
- Keep the detail routes on the same shared studio query cache as the index so save/publish/cover changes stay client-fast without route refreshes.
- For external-project entries that need strong visual identity, treat cover image upload/replacement/reordering as a first-class workflow instead of burying it under generic asset editing.
- For external-project collections that need separate operator titles and site-nav labels, keep the operator-facing title on `collection.title` and store the external nav copy in `collection.config.navigation.title`.
- For first external integrations, replace local hardcoded content registries with a single server adapter over the EPM delivery payload, then preserve the existing page design by passing mapped props into client components.
- For external branded sites, never ship public-facing copy like “live EPM delivery” or other integration explanations. Treat EPM singleton sections and profile data as the site’s public content source, and keep operational/delivery status language inside the admin product only.
- If a published SDK package is intended for external Next/Bun apps, verify it resolves through built `dist` declarations and runtime files before asking downstream apps to integrate it. Source-based package entrypoints can leak internal monorepo typings into consumers.
- Mount Vercel Analytics and Speed Insights only on actual Vercel deployments. Self-hosted environments should not emit `/_vercel/*` requests that immediately `404`.
- Add any new user-facing copy to both `apps/web/messages/en.json` and `apps/web/messages/vi.json`, then run `bun i18n:sort`.
