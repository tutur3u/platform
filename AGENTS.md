# AGENTS.md – Operating Manual for Autonomous AI Assistants

## 1. Purpose & Audience

Foundational mandates here take absolute precedence. **NEVER** invent ad-hoc behavior.

## 2. Canonical Guardrails & Hard Boundaries

### 2.1 Hard Prohibitions (NEVER DO)

- **Long-Running / Build Commands**: NEVER run `bun dev`, `bun run build`, `bun build`, or equivalent build/compile/bundling operations unless the user **explicitly requests** it.
- **Supabase Production Push**: NEVER run `bun sb:push` or `bun sb:linkpush`. Prepare migrations; the user applies them.
- **Auto-Fixing & Verification**: assistants may run `bun type-check`, `bun check`, `bun format`, and `bun ff` when (a) the user explicitly requests it, or (b) a workspace playbook requires it for files you changed.
- **Sensitive Data**: NEVER commit secrets, API keys, tokens, or credentials. Reference environment variables by name only.
- **Manual Dependency Edits**: NEVER manually edit `package.json` to add or update dependencies. Always use the CLI.
- **UI Antipatterns**: NEVER use native browser dialogs (alert, confirm), hard-coded color classes (use dynamic-*), or emojis in UI code (use lucide-react via @tuturuuu/icons).
- **Data Fetching/Mutation (#1 Violation)**: **NEVER use useEffect for data fetching.** TanStack Query (useQuery/useMutation) is the **mandatory** standard. Raw fetch() in client components is forbidden.
- **Git Naming Conventions**: Use Conventional Commits for authored commits and conventional branch names that match the repo checker (`feature/`, `feat/`, `fix/`, `bugfix/`, `hotfix/`, `release/`, `chore/`, `docs/`, `style/`, `refactor/`, `perf/`, `dependabot/`, `claude/`). Git-generated merge commits may keep their standard merge subject.

### 2.2 Mandatory Actions

- **Type Integrity**: Always run `bun sb:typegen` after schema changes. **Prefer** importing types from `packages/types/src/db.ts`.
- **Bilingual Support**: ALWAYS provide translations for both English (en.json) AND Vietnamese (vi.json) for all user-facing strings.
- **Shared UI Translation Parity**: When adding new translation keys consumed from shared packages like `packages/ui` (for example `common.*` keys), update every app-level `messages/en.json` and `messages/vi.json` bundle that ships that shared UI, and keep the message files alphabetically sorted.
- **Translation Sorting Follow-Through**: After editing any `messages/en.json` or `messages/vi.json` file, run `bun i18n:sort` before the final `bun check`. `bun check` enforces alphabetical ordering and will fail even when key parity is otherwise correct.
- **Navigation Parity**: ALWAYS update `navigation.tsx` in the relevant app when adding new routes (aliases + children + icons + permissions).
- **Proactive Refactoring**: Evaluate files >400 LOC and components >200 LOC for extraction into smaller, focused units.
- **Unified Verification**: If your changes touch TypeScript/Javascript files (or root scripts/config that affect repo-wide checks), end your session with `bun check`.
- **Flutter L10n Regeneration**: After adding or renaming keys in `apps/mobile/lib/l10n/arb/*.arb`, run `flutter gen-l10n` before `flutter analyze` or tests. The generated `app_localizations*.dart` files are tracked here, and stale generated code will surface as undefined localization getters even when the ARB files are correct.
- **UI Preflight Hygiene**: For newly added/edited files, format files with `bun ff` before running verification checks to avoid `biome` issues.
- **Formatting Workflow**: For fixing formatting issues, try `bun ff` first before making manual edits.
- **Documentation Follow-Through**: At the end of every session, document durable operational, architectural, deployment, or workflow knowledge in `apps/docs` whenever the work changes how the team should build, run, debug, deploy, or operate the system. Do not stop at code changes alone.
- **Session Retrospective**: Conduct a retrospective at the end of every session to document mistakes and update these guidelines.

## 3. Repository Structure & Semantics

### 3.1 Monorepo Layout

- **`apps/web`**: Main platform (Next.js 16 App Router, Port 7803).
- **`apps/mobile`**: Flutter mobile app (iOS/Android, BLoC/Cubit).
- **`apps/database`**: Supabase migrations and configuration.
- **`apps/discord`**: Python Discord utilities.
- **`packages/*`**: Shared logic (ui, ai, types, utils, supabase, auth, payment).

### 3.2 Semantics

- **Workspace Protocol**: Internal packages use `workspace:*`.
- **Server Components**: Default to Server Components; use 'use client' only when state/interactivity is required.
- **Satellite App API Ownership**: Dedicated satellite apps like `apps/cms` may own UI, routing, and auth shells, but protected product APIs should remain centralized in `apps/web` behind its proxy, bot-protection, and infra layer. Satellite apps should forward `/api/*` to `apps/web` and consume those routes through `packages/internal-api`.
- **Type Inference**: Import extended types from `@tuturuuu/types/db`. Never hand-edit generated type files. Refrain from ad-hoc type definitions that duplicate DB types; extend them in `packages/types` if necessary.
- **Shared Types Export Parity**: When adding new modules under `packages/types/src/primitives`, keep `packages/types/package.json` exports and the corresponding `dist/primitives/*` entries in sync before running app tests. Vitest/Vite resolves `@tuturuuu/types` through package exports, not raw source files.
- **Zod Runtime Enums Need Stable Runtime Tuples**: When a web schema uses `z.enum(...)` for newly added shared-type unions, do not assume package-exported runtime arrays will be fresh everywhere in the same patch. Prefer a stable local tuple in the schema or verify the exported runtime value is rebuilt and resolvable before relying on it in tests.
- **Shared DB Row Aliases**: If a query row shape is reused beyond a single inline callback, define/export it from `packages/types/src/db.ts` (and consume it through `@tuturuuu/types`) instead of recreating local `Pick<Database...>` aliases. For type-only Supabase usage, import `TypedSupabaseClient` from `@tuturuuu/supabase/types`, not `@tuturuuu/supabase/next/client`.

## 4. Canonical Workflows

### 4.1 Database Migrations

1. Create: `bun sb:new` -> Edit SQL (prefer additive).
2. Local Test: `bun sb:up` (agent-allowed for local Docker instance).
3. Typegen: Incorporate types from `packages/types` after user runs `bun sb:typegen`.
4. Usage: Use `normalizeWorkspaceId(wsId)` in API routes to handle "personal" vs UUID.

### 4.2 UI & Navigation

1. **Translations**: Add keys to `en.json` AND `vi.json`.
2. **Navigation**: Add route to `aliases` and `children` in `apps/web/src/app/[locale]/(dashboard)/[wsId]/navigation.tsx`.
3. **Icons**: Use `@tuturuuu/icons` (Lucide wrapper).
4. **Dialogs**: Use `@tuturuuu/ui/dialog`.

### 4.3 Settings Centralization

- **Shell**: Use `SettingsDialogShell` from `@tuturuuu/ui/custom/settings-dialog-shell`.
- **Location**: Add tabs to the app's `SettingsDialog`. **NEVER** create separate settings pages.
- **Extraction**: Extract portable settings to `packages/ui` if they have no @/ imports.

### 4.4 Documentation

- Changes that introduce or materially change team-facing knowledge MUST be reflected in `apps/docs` within the same session.
- Prefer updating an existing runbook/page when the knowledge belongs there; create a new page only when the topic does not fit the current docs structure.
- New pages MUST be added to `apps/docs/docs.json` or they will not be visible.

### 4.5 Adding Dependencies

- **Method**: To add a new package to `apps/web`: `cd apps/web && bun add <package>`.
- **UI Package**: To add a new package to `packages/ui`: `cd packages/ui && bun add <package>`.
- **Constraint**: **NEVER** manually edit `package.json` to add a new package. Use the `bun add` command to ensure `bun.lock` remains consistent.

## 5. Engineering Standards

### 5.1 Data Fetching (TanStack Query)

- **Mandatory Wrapper**: All client-side fetching/mutation must use `useQuery`/`useMutation`.
- **HTTP Cache Bypass**: Every { fetch } inside a `queryFn` MUST include { cache: 'no-store' }.
- **Realtime (Kanban)**: Use **Supabase Broadcast** via `BoardBroadcastContext`. **NEVER** invalidate queries or use `postgres_changes` for task sync in boards.
- **Internal API First**: When client or shared UI code needs app API access, add or extend helpers in `packages/internal-api/src/*` and consume those helpers instead of scattering raw `fetch('/api/...')` calls through hooks/components. Treat `packages/internal-api` as the canonical client boundary for app APIs.
- **Interactive Dashboard Pages Should Use Thin Server Gates**: For dense dashboard surfaces that need client-side search, sorting, pagination, or folder-style navigation, keep the route page server-side only for auth/permission gating and move all view data into a client shell backed by TanStack Query, `@tuturuuu/internal-api`, and `nuqs`. Do not fall back to `router.refresh()` for routine explorer/list updates; invalidate query keys instead.
- **Supabase Browser Client Deprecation Migration**: When `@tuturuuu/supabase/next/client` or `createDynamicClient` is flagged as deprecated in app code, do not swap to temporary browser clients (`auth-browser`/`realtime-browser`) for CRUD or storage paths. Migrate the call site to an authenticated Internal API route and consume it through `@tuturuuu/internal-api` helpers.
- **Internal API First for App CRUD**: When app code needs authenticated workspace/user CRUD, add or extend the route and consume it through `packages/internal-api/src/*` instead of scattering raw `fetch('/api/...')` calls across components. Treat `packages/internal-api` as the canonical client contract for internal APIs.
- **Create/Update Route Schemas Must Share Nullability Semantics**: When the client and internal API intentionally use `null` for optional persisted fields (for example `subtitle`, `summary`, or `scheduled_for`), keep `POST` and `PATCH` route validators aligned. Do not let create routes accept only `undefined` while update routes accept `null`, or new-record flows will fail even though edit flows work.
- **Two-Step AI Preview/Create Routes Must Persist Reviewed Payloads Without Re-Running AI**: For flows that first preview AI output and then save user-reviewed objects (for example task journal capture), treat the save request as a plain persistence path once validated `tasks` are present. Do not re-resolve models, re-check credits, or regenerate content during the save step, or reviewed output can fail to persist after a successful preview.
- **Workspace Destination Pickers Must Clear Dependent IDs Synchronously**: In task/location pickers where a workspace selection controls the valid boards/lists beneath it, clear the selected board/list immediately in the same change handler when the workspace changes, and clear the selected list when the board changes. Do not rely only on later effects to prune stale IDs, or the UI can submit mismatched workspace/list pairs before the derived options refresh.
- **Multi-Step Create Flows Must Clear Draft State Only After The Persist Call Succeeds**: When a user reviews/generated content, chooses a destination, and then submits a create mutation, keep the confirmed payload and current step alive until the server responds successfully. If the mutation fails, reopen the retry surface with the draft still attached instead of clearing review state optimistically.
- **Workspace-Scoped Companion Routes Must Normalize IDs The Same Way**: If one route in a feature accepts workspace aliases such as `personal` and another route in the same flow verifies membership or child resources, normalize the workspace ID in both routes before access checks. Do not let selector/bootstrap routes use normalized IDs while the final save route still validates against the raw URL segment.
- **Workspace-Scoped Child Resource Writes Must Verify Membership Before Admin Lookups**: If a route uses an admin-backed lookup to validate a child resource such as a task list, board, or project, first verify that the caller is a member of the target workspace with the request-scoped client. Only then resolve the child resource against the normalized workspace ID. Do not let admin-backed child lookups become a substitute for workspace access checks.
- **Post-Membership Workspace Validation Should Stay On The Same Authorized Admin Path**: Once a workspace-scoped route has verified membership with the request-scoped client, keep protected follow-up validation reads for child resources like `task_projects` on the same admin-backed, workspace-scoped path as list validation. Mixing an admin-backed list check with caller-scoped project-table reads can reintroduce RLS failures even though access was already proven.
- **Post-Membership Workspace Writes Should Use The Same Authorized Admin Path**: After a workspace-scoped route proves membership with the request-scoped client, keep protected inserts and relation-table writes on the same admin-backed path as the validated child-resource lookups, and stamp actor-owned columns explicitly. Mixing admin-backed validation with caller-scoped writes can still fail on tables like `tasks` even though access was already established.
- **Studio Preview Surfaces Must Reuse Preview Delivery Routes**: For authoring experiences that expose a “live preview” of published or previewable content (for example external-project studios), keep the preview source of truth on the existing preview delivery/API route and fetch it with TanStack Query. Local draft rendering may exist for immediacy, but do not introduce a second client-only payload contract that can drift from the delivered preview.
- **External Branded Sites Must Read Public Copy From EPM Content, Not Integration Banners**: When wiring an external app like Yoola to EPM, do not leave public-facing copy that explains the integration itself (for example “live EPM delivery” status text). Drive hero/about/writing/gallery copy from `singleton-sections` entries plus `profile_data`, and keep operational language inside the admin workspace only.
- **EPM Description Fields Should Stay Markdown-Ready For External Delivery**: If an external app consumes the persisted `summary` field as a description or excerpt, present it in EPM as a markdown-capable description editor rather than a plain “summary” textarea. Keep longform page or writing content in markdown blocks so short and long copy share one delivery contract.
- **Preview-And-Admin Products Need Explicit Modes, Not Hybrid Dashboards**: When one tool must act as both a served-content preview and an operations console, default to an explicit `Preview / Edit` split instead of layering workflow cards, summaries, and admin chrome on top of the visual preview. The preview surface should feel like the delivered product; the edit surface should stay focused on routing and operations.
- **Large Client Admin Surfaces Should Split Query Shells From Render Modules**: When a client-side admin route grows beyond a few hundred lines, keep the main file focused on TanStack Query state, mutations, and routing, then extract helpers, skeletons, dialogs, and per-mode/per-section render components into adjacent files. Do not let one mega TSX file own every utility and render branch.
- **Visual Content Indexes Should Use One Gallery Surface And Route Depth Elsewhere**: For visual content tools like EPM, do not spend the main index width on a browser column plus a second preview column. Use the full space for a grid/gallery and route into a dedicated near-fullscreen detail page for deep preview or editing.
- **Shared Studio Detail Routes Must Mutate The Same Query Cache**: For studio-style products with dedicated entry or collection detail pages, keep those routes on the same TanStack Query dataset as the index and patch/invalidate that cache on save/publish/upload instead of falling back to `router.refresh()`. Server route files should remain auth/permission gates only.
- **Dense Admin Editors Should Not Be Duplicated Inline And In Overlays**: For heavy admin authoring flows with large item forms plus previews, keep collection/workspace controls inline and move item-level editing into a dedicated page or near-fullscreen dialog. Do not maintain two simultaneously mounted copies of the same editor just to offer an “expanded” mode; duplicate controls drift, complicate refs/uploads, and make save state harder to reason about.
- **Dense Split-View Admin Pages Need One Summary Surface And Optional Preview**: In multi-column admin studios, keep the main editing column focused on compact summaries and lightweight controls. If a permanent preview column makes the workspace crowded, move preview behind an explicit action such as a drawer or sheet instead of squeezing three dense columns onto the page. Do not render another full preview treatment inside the summary column; use tooltips for secondary explanations instead of permanently visible helper copy when density is already high.
- **Protected Internal Asset URLs Must Bypass `next/image` Optimization**: If a UI renders authenticated or signed app-local asset routes such as `/api/v1/.../assets/:id`, do not feed them through the Next.js optimizer. Mark those images `unoptimized` or render them without `next/image`, or production surfaces can spam `/_next/image?... 400` even when the direct asset route works.
- **Supabase Folder Deletes Must Recurse Through The Storage API, Not `storage.objects` SQL**: For workspace Drive delete flows, do not depend on direct `schema('storage').from('objects')` scans just to enumerate a folder before removal. Hosted environments can diverge on that path; use recursive `storage.from(bucket).list(...)` traversal and then delete the collected object keys.
- **Controlled Dashboard Tabs Need Testable Initial State**: For complex `apps/web` admin dashboards built around controlled tabs, expose an optional initial-tab/state prop or equivalent deterministic entry point for tests instead of relying on tab-click transitions in Vitest/JSDOM. This keeps interaction tests stable without changing the default runtime tab.
- **Repeated Action Labels In Dashboard Tests Need Scoped Queries**: If a redesigned dashboard intentionally repeats actions such as `Open preview` in both hero and toolbar zones, scope Vitest Testing Library queries with `within(...)` or a section/test id instead of relying on a globally unique accessible name.
- **API-Only Table Parity Across Surfaces**: If a mobile or web feature already has an API-backed repository/helper for a protected table, reuse that path for dashboard/summary/home views too. Do not mix API-backed list screens with direct Supabase table reads for the same entity, or protected-table permission errors will appear only on some surfaces.
- **Shared Route Helpers Must Forward Request Auth**: When extracting server-side data loaders that are reused by both Server Actions and Route Handlers, make the helper accept an optional `request` and pass it into `createClient(request)` whenever a route invokes it. Helpers that always call cookie-only `createClient()` will work on the web but return `401` for mobile/API callers sending Bearer tokens.
- **Mira Attachment Tools Must Resolve Current-Chat Files By Name**: Chat uploads can start under a temp path and then move into `{wsId}/chats/ai/resources/{chatId}` before tool execution, while model tool calls may pass the displayed filename in `storagePath`. File conversion tools should resolve bare filenames or same-workspace stale attachment paths against the current chat folder, while still rejecting cross-workspace full paths.
- **`bun ff -- <files>` Still Triggers Repo-Wide Biome Passes**: The root `ff` script runs `biome format --write && biome lint --write && biome check --write` before any forwarded file arguments reach the final check command. Expect unrelated schema-version infos or sandboxed `.agents` permission noise during formatting; use `bun ff` for touched files as required, then treat `bun check` as the authoritative verification signal for the workspace.
- **ANSI Strip Helpers Must Stay `new RegExp(...)`-Based In Root Scripts**: Biome can rewrite `\u001B` regex literals in root Node scripts into a form that then fails `bun check` under `noControlCharactersInRegex`. If you need to strip ANSI codes in scripts like `scripts/watch-blue-green-deploy.js`, keep the implementation in the `new RegExp('\\u001B\\[[0-9;]*m', 'g')` form and re-verify it after `bun ff`.
- **Literal `${...}` Source Checks In Root Scripts Should Use Regex, Not Plain Strings**: When a root validation script needs to assert that another source file contains a literal template placeholder such as `${process.env.PORT || 7803}`, do not embed that token inside a normal string literal. Biome flags it with `lint/suspicious/noTemplateCurlyInString`, so express the match as a regex (or another escaped form) and cover it with the corresponding script test.
- **Blue/Green Proxy Verification Must Use Internal Runtime Paths, Not Public App Health Routes**: Deployment checks that run through `web-proxy` should target internal request-tracker endpoints like `/__platform/drain-status` rather than public paths such as `/api/health`. Public routes can hit proxy or middleware rate limits and produce false `429` deployment failures even when the new color is healthy.
- **Long-Running Watchers Need Explicit Lock-Recovery Modes**: If a long-running local watcher uses a PID lock, expose explicit conflict actions instead of a dead-end failure. For the blue/green watcher, support a read-only resume/mirror mode and a replace mode that terminates the existing PID before taking over, and document both in `apps/docs`.
- **Blue/Green Auto-Deploy Watcher Must Refresh Host Bun State After Pulls**: When the blue/green watcher fast-forwards to a new commit, run `bun upgrade` and then `bun i` before the deploy handoff continues. If those host-side install steps leave only `bun.lock` dirty, keep the watcher polling instead of treating that file as a blocking dirty worktree. Recoverable Git command failures should not terminate the watcher; retry after 1 minute and back off exponentially on consecutive failures.
- **`apps/web` Package Tests Use Vitest Discovery, Not `node:test`**: If you add test files under `apps/web`, write them as Vitest suites using `import { describe, it, expect } from 'vitest'`. Plain `node:test` scripts with `.test.js` names are picked up by the package test runner and will fail `bun check` with "No test suite found" or CommonJS/Vitest import errors.
- **BuildKit Throttling Needs A Dedicated Builder, Not `docker compose build --memory`**: Docker documents that `docker compose build --memory` is not supported by BuildKit. For Dockerized web builds on shared machines, use a dedicated `docker-container` Buildx builder with driver-level memory / CPU limits plus BuildKit `max-parallelism`, and point the workflow at it with `BUILDX_BUILDER`.
- **Git Auto-Deploy Watchers Must Lock Branch And Upstream At Startup**: For long-running deploy pollers, capture the current branch plus tracked upstream once, write a PID-backed lock file, use `git pull --ff-only`, and stop if the checked-out branch changes. Dirty or diverged worktrees should log and skip instead of forcing merges.

## 6. Known Gotchas & Patterns

### 6.1 Database & Supabase

- **PostgREST URL Limit**: Use .eq() or batching instead of .in() with >1000 IDs to avoid 8KB proxy limits.
- **Service-Role Approval Writes**: If a route must write with `createAdminClient` because RLS can block the mutation, fetch the caller with `createClient(request)` first and stamp any actor-owned columns explicitly (for example `approved_by`, `rejected_by`, `updated_by`) instead of relying on triggers or policies that read `auth.uid()`. Keep read-side verification on `sbAdmin`, but always scope those reads with explicit workspace/resource checks before mutating.
- **Large Workspace-Scoped ID Lists Need Join Tables, Not CSV Config Blobs**: Do not store growing sets such as workspace user-group selections in `workspace_configs.value` as comma-separated IDs. The strict text-limit trigger on generic config values will eventually reject long lists, and diffing/querying blobs is inefficient. Use a dedicated workspace-scoped join table and special-case the config API if the client contract must stay string-based.
- **User Email**: Never query `public.users.email` (it doesn't exist). Use `public.user_private_details`.
- **Reset-Only Local Defaults**: If behavior should apply only after `bun sb:reset` (local dev bootstrap), implement it in `apps/database/scripts/*` and wire it into the reset script; do not encode reset-only behavior in migrations.
- **Strict Text-Limit Rollouts**: When tightening DB text/octet limits, audit `sb:reset` seed/bootstrap scripts and external metadata syncs in the same patch. Local resets must keep working under the new ceilings, or typegen/pgTAP verification will give false confidence.
- **Shared Limit Parity**: When changing a field limit that is enforced in both app code and Supabase migrations, update every enforcement layer in the same patch: shared constants, API validation schemas, `strict_text_field_char_limit` helpers, trigger byte limits, and explicit table `CHECK` constraints. Limit drift between these layers can pass local app checks while failing production migration pushes.
- **Seed-Dependent Migration Backfills**: When a migration backfills foreign keys to seeded/reference tables, do not assume those rows already exist in every local/reset state. Prefer nullable columns or conditional subqueries during backfill, and keep runtime/API fallbacks authoritative until the reference data is present.
- **Table-Replacement Migrations Must Materialize Target Rows Before Rewiring Children**: When replacing one parent table with another (for example moving child rows from `course_id` to `group_id`), first copy or upsert the source rows into the target table, then backfill every dependent child foreign key, validate that no `NULL` or orphaned values remain, and only then drop old constraints, columns, indexes, or tables. Dropping the source relation first can hide data behind new RLS paths and break uniqueness assumptions.
- **Supabase Typegen Requires Applied Migrations**: After adding or changing a Supabase migration that affects schema types, apply it to the local database with `bun sb:up` before running `bun sb:typegen`. The generator reads the live local schema, not unapplied SQL files, so skipping the local apply step leaves generated types stale.
- **PR Migration Filenames Must Stay Ahead Of `main`**: After merging or rebasing `main`, re-check any new files under `apps/database/supabase/migrations`. If a PR migration timestamp now sorts before the latest migration on `main`, rename it forward before pushing or `check-migrations` will fail even when the SQL itself is correct.
- **Supabase Typegen Redirect Truncation**: Before running `bun sb:typegen`, confirm the local Docker/Supabase stack is actually reachable. The script redirects generator output straight into `packages/types/src/supabase.ts`, so if the generator fails before producing content, the file can be truncated to zero bytes and break unrelated type-checks until it is restored.
- **Supabase Helper ReturnType Can Erase Table Inference**: Do not define helper client aliases as `Awaited<ReturnType<typeof createAdminClient>>` or similar generic `ReturnType` wrappers around Supabase client factories. That loses the concrete `Database` binding in some call sites and turns `.from(...)` rows into `never`. Use `TypedSupabaseClient` (or cast the awaited helper result to it) for shared admin/request client parameters instead.
- **Admin Routes Must Not Use Service-Role Clients For `auth.uid()`-Guarded RPCs**: If a `SECURITY DEFINER` RPC still validates `auth.uid()` internally, call it with the authenticated request-scoped Supabase client after permission checks, not `createAdminClient()`. Service-role clients bypass RLS but do not carry the caller session, so these RPCs fail with `Unauthorized` even for properly authorized admins.
- **Trigger Hardening Must Rewrite DDL Checks Too**: If you tighten text or payload limits with a trigger/helper migration, also rebuild the underlying `CHECK` constraints for those columns. Leaving legacy `char_length`/`octet_length` constraints in place means Supabase table definitions still advertise and enforce stale loose limits even though the trigger looks strict.
- **AI Gateway Model Text Is External Catalog Metadata**: `ai_gateway_models` syncs provider-owned public catalog fields whose descriptions and names can exceed Tuturuuu's generic public text ceilings. Keep its text columns exempt from strict text-field length triggers and generated text `CHECK` constraints, while preserving JSON payload-size checks for pricing metadata.
- **Workspace-Member Cleanup Triggers Need SECURITY DEFINER**: If a `workspace_members` trigger updates another protected workspace table (for example clearing `task_projects.lead_id` on member delete), define the trigger function with `SECURITY DEFINER` and `SET search_path = public`. Invoker-security trigger functions can fail with `permission denied` during member removal even when the outer delete itself is allowed.
- **Workspace-Scoped Integration Tables Need DB-Level Workspace Parity Checks**: If an integration child table stores its own `ws_id` while also referencing workspace-owned parents (for example SePay wallet links, webhook endpoints, or webhook events), enforce matching workspace ids with trigger-based validation or an equivalent database constraint. Do not rely on route/service code alone to keep those foreign keys aligned.
- **Workspace-Scoped Mutation Follow-Through**: For workspace-scoped `UPDATE`/`DELETE` API handlers, request returning rows (`select(...).maybeSingle()` or equivalent) and stop follow-up side effects when no row matched; never run dependent child-table deletes/updates based only on a requested ID.
- **Membership Check Error Parity**: In API routes that gate access with `workspace_members` lookups, always handle query `error` explicitly and return `500` for membership lookup failures; return `403` only when the lookup succeeds but no membership row exists.
- **Signed Upload Preview Timing**: For Supabase storage uploads using `createSignedUploadUrl`, do not request a signed read/preview URL before the client uploads the object. The file does not exist yet and storage will return object-not-found errors. Use a local blob preview immediately after upload in the client, and resolve signed read URLs only after the object exists.
- **Signed Upload URL Creation For Protected Buckets Must Split Auth And Storage Clients**: In web/API routes, keep workspace/user authorization checks on the request-scoped Supabase client, but generate Supabase Storage signed upload URLs with `createDynamicAdminClient()` (or another admin-backed storage client). `createSignedUploadUrl` inserts storage metadata and can fail with `new row violates row-level security policy` when invoked through the caller-scoped client even after the route has already verified access.
- **Mixed Storage Signed Upload Retries Must Preserve Auth When Present**: In shared upload helpers that support both Supabase signed uploads and S3-compatible presigned uploads, keep the `Authorization` header on retry attempts whenever the server issued a bearer token and only relax headers that commonly break presigned PUTs (for example `Content-Type`). Supabase signed upload URLs still require the bearer token on every attempt, while R2-style presigned uploads must also work when no token is returned at all.
- **Workspace Storage Backend Migrations Must Use Explicit Source/Target Backends**: When copying Drive data between Supabase storage and Cloudflare R2, do not reuse helpers that implicitly resolve the workspace's active provider or active-provider quota totals. Migrations must read and write against the requested source/target backends directly and avoid double-counting the same logical workspace inventory against quota enforcement during the copy.
- **Proxy-Locked Table RPCs**: When a public table has direct privileges revoked to force API-only access, do not leave write/read RPCs as `SECURITY INVOKER`. Use `SECURITY DEFINER` with `SET search_path = public` and re-check membership/permissions inside the function, or authenticated callers will fail with table-permission errors while service-role callers lose `auth.uid()`-based authorization context.
- **Authenticated Views Need `security_invoker` Before Broad Grants**: If a Postgres view reads RLS-protected tables and you grant `SELECT` on that view to `authenticated`, declare the view with `WITH (security_invoker = true)` or `ALTER VIEW ... SET (security_invoker = true)` in the same migration chain. Owner-privileged views can bypass the underlying RLS checks and leak cross-workspace data.
- **RPC Signature Changes Must Drop Legacy Overloads**: When adding parameters to a PostgREST-exposed RPC, do not rely on `create or replace function` to "upgrade" the old signature. PostgreSQL treats the new parameter list as a new overload, and PostgREST named-argument RPC calls can become ambiguous when omitted optional params still match both signatures. Drop the legacy function signature in the same migration that introduces the new one.
- **Dashboard Queue Status Must Reflect Real Queue Rows**: For post-email dashboards or similar observability RPCs, do not synthesize `queued` from "no queue row and no sent email". Only actual queue rows (or a persisted sent email) should drive delivery status, or the UI can show stale queued counts long after the worker has drained the queue.
- **Post Email Delivery Recovery Must Use `email_audit` As The Durable Ledger**: In the post email queue, provider success is durably recorded in `email_audit` before `sent_emails` backfill completes. Retries and stale-row recovery must consult matching sent audit rows for the same post recipient and backfill `sent_emails` opportunistically instead of resending the email just because local persistence failed.
- **Queue Workers Must Claim Only What They Can Start**: For the post email queue and similar workers, do not front-claim the whole `sendLimit` before processing. Claim only the slice that can actually start within the current concurrency / time budget, or stale-processing cleanup can requeue rows that never really began and create duplicate attempts.
- **Capped Reconciliation RPCs Must Prioritize Actionable Rows Before Limiting**: When a cron or repair RPC caps work by post/entity count, rank rows that can actually be mutated (for example deliverable recipients with a resolvable sender) ahead of permanently undeliverable rows before applying the limit. Otherwise the same undeliverable slice can monopolize every run and starve later actionable records forever.
- **Paged Repair RPCs Need Caller-Controlled Offsets**: If a cron or reconciliation RPC can legitimately return zero mutations for one limited page while later pages may still contain actionable rows, expose an explicit offset/skip parameter and let the caller scan multiple pages within a time budget. One fixed "first page only" repair pass is not robust enough for large backlogs.
- **Left-Joined Diagnostics Must Distinguish "Not Selected" From "Missing"**: In SQL diagnostics over a limited subset, do not count `left join` nulls as domain-level missing data unless the row was explicitly selected into the subset. Aggregate subset diagnostics from the subset CTE itself, or unmatched outer rows will inflate counters like `missing_user_record`.
- **Malformed Supabase SSR Auth Cookies Must Be Sanitized And Counted As Edge Abuse**: If an invalid `sb-<project-ref>-auth-token` cookie reaches `@supabase/ssr`, Base64-URL decoding can throw before route-level auth checks or middleware redirects run. Sanitize and clear malformed `base64-` Supabase auth cookies in `packages/supabase` cookie adapters, and reject them in the web API proxy before rate-limit/auth work. Repeated malformed-cookie hits from the same IP should increment an edge-side abuse counter and escalate into an IP block so bad traffic stops before it reaches route handlers.
- **Self-Hosted Blue/Green Promotions Must Keep A Warm Standby Online**: For the Dockerized `apps/web` production flow behind Cloudflare Tunnel or other long-lived upstream connections, do not retire the old color immediately after promotion. Keep the demoted color alive as the nginx backup upstream, drain its in-flight requests first, and increase nginx upstream response-header buffers for Supabase auth flows so normal browsers do not hit `502 Host Error` or `upstream sent too big header` only after a rollout.
- **Backend Auth/DB 429s Must Cascade Into Proxy IP Blocks And Auto-Suspend Verified Abusers**: If a Supabase auth call or downstream database mutation returns a backend `429`, do not just relay the error. Immediately seed the proxy-visible edge IP block cache so subsequent requests are stopped before they reach the backend, and auto-suspend the authenticated user when the route already has a verified `user.id`. Anonymous or replayed-token floods may only have a trustworthy IP signal; in those cases block the IP and do not infer a user identity from unverified tokens or cookies.
- **Proxy Suspicious-Request Heuristics Must Not Shadow Canonical Payload Limits**: In `apps/web/src/proxy.ts`, anonymous-abuse checks may reject malformed headers, scanner paths, or query spam early, but they must not turn oversized request bodies into generic `400` responses before the downstream API proxy guard can return its canonical `413 Payload Too Large`. Keep body-size enforcement in the shared payload guard, and cover the precedence with a regression test when proxy abuse logic changes.
- **Auth-Sensitive Time Tracking Request Edits Need Session Writes And Admin Cleanup Fallback**: For `time_tracking_requests` content edits, keep the actual row update on the authenticated request-scoped Supabase client so triggers/RPCs that depend on `auth.uid()` continue to work. If the flow can leave behind pre-uploaded storage objects, use an admin-backed cleanup fallback for storage deletes so a failed DB write does not strand uploads behind storage RLS errors.
- **Attendance Export Ranges Should Stay String-Inclusive And Paginated**: For workspace attendance exports backed by `user_group_attendance`, filter the persisted `date` values with inclusive `gte(startDate)` and `lte(endDate)` string bounds instead of converting to timezone-sensitive timestamps. Always page the export query; Supabase's default row limits are too small for "export everyone" flows.

### 6.2 UI & Rendering Patterns

- **Group-Scoped Admin Selectors Should Default-Load And Auto-Select**: For dashboard tools that need a workspace user group before the main content can render, do not hide the first useful state behind an empty search box. Fetch the first page by default, auto-select the first available group when there is no explicit selection yet, and support incremental/infinite loading inside the dropdown for large group lists.
- **Task Board Closed Statuses Are Multi-List**: Treat `closed` as a status bucket, not a singleton column. When changing task-board list management, update every create/edit/reorder surface together and keep the database/API layer in sync (kanban add-list UI, board layout settings, grouped-board drag rules, shared list dialogs/actions, task-board list routes, RPCs, template import paths, and any DB triggers) so multiple terminal lists such as `Abandoned`, `Not Planned`, or `Duplicate` remain supported consistently.
- **Task Title Limits Must Use The Domain Constant End-To-End**: Task titles are not generic `name` fields. When changing task title length, update `MAX_TASK_NAME_LENGTH`, every task-only API/schema path (`tasks`, shared task editing, task drafts, quick-journal task extraction, relationship creation), and every database enforcement layer (`strict_text_field_char_limit`, explicit `tasks` check constraints, and DB tests) in the same patch.
- **Task Dialog Limit Warnings Must Validate In The Save Path Too**: Do not rely only on input `maxLength` for task titles or descriptions. Task dialogs can be hydrated from drafts, relationship flows, or other programmatic state updates, so the final save hook must enforce the same limits and surface the warning there before any mutation runs.
- **Next-Themes Provider Ordering On Next.js 16**: In `apps/web`, keep `next-themes` `ThemeProvider` as the outermost provider boundary around other client providers (for example tRPC or `NextIntlClientProvider`). If `ThemeProvider` is nested inside another client provider, its inline theme script can be rendered on the client and Next.js 16 will throw `Encountered a script tag while rendering React component`.
- **Self-Hosted Web Layouts Must Gate Vercel Client Telemetry**: In `apps/web` layouts, mount `@vercel/analytics` and `@vercel/speed-insights` only when the runtime is actually on Vercel (`VERCEL` / `VERCEL_URL`). Leaving them on for self-hosted deployments creates noisy `/_vercel/insights/view` 404s without providing telemetry.
- **Next.js 16 Route Proxies Must Use Direct Undici Calls For Custom Dispatchers**: In `apps/web` route handlers, do not pass a custom Undici `dispatcher` into the global `fetch`. Next.js' wrapped server `fetch` can fail with `UND_ERR_INVALID_ARG` / `invalid onRequestStart method`. If a proxy route needs transport-level Undici options such as a larger `maxHeaderSize`, call `undici.request(...)` directly and attach the dispatcher there.
- **Rich Text Editor Isolated Roots Must Not Mount `next-themes` Providers**: In `packages/ui` editor node views or any other `createRoot(...)` islands, do not wrap the isolated subtree in `next-themes` `ThemeProvider`. That provider injects an inline script, which Next.js 16 will reject when it is rendered from a client subtree. Resolve theme from the DOM (`document.documentElement` + `prefers-color-scheme`) inside the isolated root instead.
- **YouTube Embeds Need A Referrer Policy**: For every app-managed YouTube `iframe` or Tiptap YouTube node, set `referrerPolicy="strict-origin-when-cross-origin"` (or `referrerpolicy` in serialized HTML attributes). Without it, embedded videos can fail with YouTube `Error 153` because the player sees no usable referrer.
- **html2canvas Exports Need Inline Brand Marks And Solid Backgrounds**: When a web preview is exported with `html2canvas`/`html2canvas-pro`, do not rely on external SVG `<img>` assets for critical branding and do not leave the export background transparent by default. Inline simple SVG marks (or rasterize them) and pass the preview's computed background color into the export call to avoid dropped logos and stray edge artifacts in downloaded images.
- **Task Deep-Link Recovery Must Use Workspace Task APIs**: For canonical task URLs (`/{wsId}/tasks/boards/{boardId}?task={taskId}`) and legacy fallback URLs (`/{wsId}/tasks/{taskId}`), resolve the task through the workspace-scoped task endpoint (`/api/v1/workspaces/{wsId}/tasks/{taskId}`), not `/api/v1/users/me/tasks/{taskId}`. The current-user endpoint is for dialog hydration and can return `404` for valid workspace task routes, including `personal`.
- **Task Deep Links Must Use The Board Route Contract**: For dashboard task dialogs and share/copy flows, use `/{wsId}/tasks/boards/{boardId}?task={taskId}` as the canonical reloadable task URL. The modal-only `/{wsId}/tasks/{taskId}` path is not a safe substitute for copied links or fake task URLs in board contexts.
- **Task Board Recycle-Bin Batch Mutations Must Reload Once**: In `apps/mobile`, bulk restore/delete actions from the task-board recycle bin must not call cubit mutations that trigger a full board reload per task. Use a non-reloading mutation path inside the loop, keep local drawer state in sync per item, then perform one final `cubit.reload()` after the batch completes.
- **Recycle-Bin Refreshes Must Drop Stale Task Selections**: When `apps/mobile` recycle-bin views reload deleted tasks after refresh, restore, or recovery from a failed batch mutation, always filter the selected task id set against the newly loaded task ids. Do not leave bulk-action state pointing at tasks that no longer exist in the current deleted-task snapshot.
- **Task Editor Delete Busy State Must Cover The Confirmation Mutation**: In `apps/mobile` task-board editors, set `_isDeleting` before opening the async delete confirmation dialog and clear it only after the dialog finishes. The editor busy state must cover the actual in-flight delete mutation, not only the post-success toast/close sequence.
- **Task Board Assignee Avatars Must Match Their Occupied Width**: In `apps/mobile` task board cards, size overlapping assignee avatar stacks from the actual occupied width (`20px` first avatar, `14px` per additional overlap, plus overflow badge width when present) instead of a fixed extra slot. Over-wide stacks create stray trailing gaps; under-sized stacks can clip the overflow badge.
- **Avatar Fallback Initials Must Not Overlay Loaded Images**: In `apps/mobile`, when a `CircleAvatar` has a valid `backgroundImage`, do not also render the fallback initials as `child`. Keep the fallback only for empty/missing avatar URLs, or initials will draw on top of successfully loaded profile images.
- **Flutter Mobile Overlay Wrapper Mandate**: In `apps/mobile`, NEVER call `shad.openDrawer`, `shad.showDialog` (for full-screen content), or `shad.showDropdown` (for filter/picker content) directly for sheet or dialog surfaces. Always use `showAdaptiveDrawer` (for content that benefits from a shadcn drawer on compact + dialog on expanded) or `showAdaptiveSheet` (for standard modal bottom sheet on compact + dialog on expanded) from `package:mobile/core/responsive/adaptive_sheet.dart`. These wrappers embed a `BackButtonListener` that intercepts the Android hardware back button and dismisses the overlay instead of triggering shell navigation. Direct shadcn overlay calls bypass this listener and cause back-press to navigate to the previous route instead of closing the overlay. Inline `shad.showDropdown` calls within an existing `showAdaptiveSheet`/`showAdaptiveDrawer` content builder (e.g. a select inside a filter sheet) are allowed. `showAdaptiveDrawer` returns a `Future<void>` so callers can optionally `await` it to run logic after the overlay closes; non-awaiting callers in async functions must wrap with `unawaited(...)`.
- **Flutter Stateful Page Logic Extraction Must Stay State-Compatible**: In `apps/mobile`, when a `StatefulWidget` page is split across `part` files, keep methods that rely on `setState`, `context`, or the page’s private fields on the concrete `State` class (or an abstract base class it extends). Do not move that logic into extensions or self-referential private mixins on the same `State` type; Dart analyzer can treat those patterns as protected-member misuse or recursive interface inheritance and break the page build.
- **Flutter Mobile Shell Action Parity**: For top-right shell actions in `apps/mobile` (for example notifications beside the avatar), default to the lighter `shad.IconButton.ghost` treatment used by section app bars unless the surrounding shell chrome already uses surfaced square triggers. If that action opens an adaptive drawer, open the drawer immediately and let the drawer content fetch/refresh its own data on mount instead of blocking the tap on a pre-open network request.
- **Finance Amount Privacy Must Be App-Scoped**: In `apps/mobile`, do not keep finance amount visibility in a single route widget. Persist it through `SettingsRepository`, expose it from a root-provided cubit, reuse the same shell action on every finance route, and point the Settings screen at that same source of truth so overview/activity/wallet/detail surfaces never drift.
- **Finance Activity Lists Need Configurable Edge Insets**: In `apps/mobile`, shared transaction accordions should not hard-code compact-mode horizontal padding. Expose a configurable inset so activity screens can render edge-to-edge on the shell surface while other finance surfaces keep their own surrounding layout.
- **Finance Activity Gutters Belong To The Page, Not The Accordion**: When removing duplicate horizontal padding from mobile finance accordions, preserve the standard screen/list padding on the parent scroll view and only zero out the accordion’s internal compact inset. Do not strip the outer page gutters just to compensate for duplicated accordion padding.
- **Finance Transfer Activity Must Collapse Mirrored Rows**: In `apps/mobile`, normalize transfer results before grouping or caching so each wallet transfer renders once per linked pair, preferring the origin-side transaction when both mirrored rows are present. Do not feed raw transfer pairs directly into activity accordions.
- **Inventory Product Draft Memory Must Stay Workspace-Scoped**: In `apps/mobile`, if you remember last-used product editor selections such as owner, product category, or linked finance category, persist them per-workspace and hydrate them only for new-product flows. Do not leak one workspace’s remembered defaults into another workspace or overwrite explicit values when editing an existing product.
- **Inventory Sell Filters Depend On Option Payload Labels**: In `apps/mobile`, if the sell/checkout UI exposes quick product-category filters, the `/api/v1/workspaces/:wsId/products/options` payload must include the human-readable product category relation/name and the mobile repository normalizer must preserve it. Otherwise the filter chips render with no options even though the UI code is present.
- **Fullscreen Finance Composers Need Explicit Centering**: In `apps/mobile`, do not rely on default tab/button primitives to keep long finance action labels centered on compact fullscreen composers. Use explicit centering and scale-down behavior for segmented mode tabs and footer action labels so text stays visually balanced instead of drifting or wrapping awkwardly.
- **Flutter Mobile Shell Action Dropdown Context**: When a shell-chrome icon action opens `shad.showDropdown` and the menu mutates route-local Bloc/Cubit or other page-scoped state, capture and use the page/root context for those mutations instead of the dropdown builder/menu-item context. Overlay contexts can lose access to the intended route subtree and make calendar/history-style mode switches fail or no-op.
- **Flutter Mobile Dropdown Menu Buttons Must Not Manually Pop Routes**: When a `shad.MenuButton` inside a dropdown already completes the intended state change, do not also call `Navigator.pop` / `maybePop` unless you truly mean to dismiss a separate route. The dropdown overlay closes on selection by itself; the extra pop can unwind the active mini-app and send the user back to `Routes.apps`.
- **Flutter Mobile Month Calendars Need Readable Cell Summaries**: When a month calendar is a first-class route in `apps/mobile`, do not reduce day cells to anonymous color bars or dots only. Keep muted spillover days visible for continuity, preserve strong today/selected states, and use compact readable event chips or labels so the month grid remains scannable without drilling into each day.
- **Flutter Mobile Month Calendars Should Use The Shell Surface**: For tab-root month calendar views in `apps/mobile`, do not wrap the entire grid in an extra rounded card or inset panel unless the design explicitly calls for a nested surface. Let the month grid live on the page’s native shell background and use cell-level states, dividers, and subtle fills for structure.
- **Flutter Mobile Month Calendars Should Fill A Stable Six-Week Grid**: On compact month views in `apps/mobile`, size the grid from the available viewport height so the full six calendar rows fill the content area predictably. Do not rely on ad-hoc child-aspect ratios that leave large dead zones or compress event indicators into an unusable strip.
- **Finance Mobile Forms Should Be Guided, Not Tabbed**: For transaction create/edit flows in `apps/mobile`, avoid splitting a small modal into in-body tabs such as “Details” and “Settings”. Prefer one scrollable composer with a live preview, grouped selector cards, and inline toggles so creation feels native and users do not miss critical fields hidden behind a second tab.
- **Finance Transaction Mode Tabs Are Allowed In Fullscreen Composers**: In `apps/mobile`, keep transaction forms as one guided scrollable composer, but it is acceptable to expose the primary `Transaction` / `Transfer` mode switch as tabs when the create/edit experience is presented fullscreen. Do not reintroduce secondary content tabs for details/settings.
- **Mobile Finance CRUD Forms Should Escalate To Fullscreen Early**: In `apps/mobile`, once finance create/edit forms accumulate multiple sections, previews, or secondary pickers (wallets, colors, icons, images), present them as fullscreen routes with sticky footer actions instead of compact modal sheets. Wallet, category, and tag editing should follow the same fullscreen scaffold as transaction composition.
- **Flutter Mobile Subtabs Belong In Shell Chrome**: When a mobile app section needs task-style sub-navigation (for example finance categories/tags, time-tracker history day/week/month, stats personal/workspace/goals, or calendar view modes), register those controls through `ShellMiniNav` and move any matching "jump to current period" affordance into `ShellChromeActions`. Do not leave the primary mode switch as an in-body segmented control once the screen is a dedicated shell route.
- **Route-Specific Search/Add Actions Belong In Shell Chrome Too**: On mobile shell-routed subsections such as time-tracker workspace stats or goals, move contextual controls like search and create actions into `ShellChromeActions` instead of repeating titled body headers, inline toolbars, or section cards. Keep the page body focused on the actual list/cards/empty state for the active sub-surface.
- **Route-Scoped Shell Registrations Must Mount Only The Active Surface**: In `apps/mobile`, do not keep multiple route-specific surfaces with their own `ShellMiniNav` / `ShellChromeActions` registrations alive together in an `IndexedStack`. For shell-routed sections like time-tracker history/stats, mount only the active surface for the current route so the injected shell chrome resolves deterministically instead of falling back to the module-level nav.
- **Stack Layouts Need A Real Sizing Child**: On Flutter mobile screens, a `Stack` whose only visible body is `Positioned.fill` can collapse to zero height if the remaining children are registration widgets or `SizedBox.shrink()`. Ensure route content includes at least one non-positioned sizing child, or render the body outside `Positioned.fill`, before assuming the surface is merely "empty."
- **Flutter Mobile Shell Surface Parity**: For tab-root pages rendered inside the shared mobile shell, keep the top chrome, page body, docked composer/footer, and bottom navigation on the same base surface color unless the design explicitly introduces a sectional card. Do not mix `scaffoldBackgroundColor` with shell chrome surfaces for assistant/chat pages, or the body will read as a mismatched panel instead of one continuous shell.
- **Fullscreen Dock Safe Area Must Stay Opaque**: For bottom-docked composers or toolbars in fullscreen mobile surfaces, do not add bottom safe-area spacing outside the dock container with transparent wrappers. Extend the dock’s own background/padding through the bottom inset so the safe area remains opaque and does not reveal scrolling content behind it.
- **Flutter Mobile Icon-Only Nav Tabs**: When a bottom or floating shell tab in `apps/mobile` hides its visible label to free space, keep the destination name in semantics and own any tap feedback animation in `ShellPage`/nav chrome rather than inside the destination page so the effect survives route transitions and remains accessible.
- **Drawer Content Must Not Recreate Drawer Chrome**: When rendering compact content inside `showAdaptiveDrawer`, treat the builder output as the drawer body only. Do not wrap it in another full-width rounded container, fake sheet background, or duplicate drag handle, or the UI will render as stacked/double sheets on mobile.
- **Anchored Overlay Height Bounds**: For custom floating overlays on medium/expanded Flutter layouts (for example avatar menus anchored to an app-bar trigger), do not use `Expanded`/`Flexible` inside the panel unless the panel itself has an explicit height. Anchored dialog cards usually only have a max-height constraint, so their content should scroll directly; reserve `Expanded` layouts for compact sheet/drawer variants that are wrapped in a bounded parent.
- **Flutter Mobile Shell Back vs Navigator Dialogs**: `ShellPage` handles Android back before GoRouter shell logic. `shad.showDialog` pushes a Flutter `DialogRoute` on the root navigator (not the shadcn drawer overlay). Before running shell back navigation, `ShellPage` pops the root navigator when `Navigator.of(context, rootNavigator: true).canPop()` is true so language/theme and other `shad.showDialog` surfaces close on back instead of changing routes. Inner `BackButtonListener` wrappers on adaptive sheets/drawers still take precedence over the shell listener.
- **Mobile Text Inputs Need A Root-Level Outside-Tap Unfocus Wrapper**: In `apps/mobile`, do not try to restore keyboard dismissal by patching individual forms with ad-hoc `GestureDetector` or per-field `onTapOutside` callbacks. Keep a single app-root pointer handler around the router/navigator that unfocuses the active input only when the tap lands outside the currently focused field so pages, dialogs, and adaptive sheets all dismiss the keyboard consistently.
- **Flutter Mobile Firebase Singletons Must Stay Lazy**: In `apps/mobile`, do not read `FirebaseMessaging.instance`, `FirebaseAuth.instance`, or similar Firebase singletons in static initializers or singleton field initializers that can run during widget tests before `Firebase.initializeApp()`. Resolve Firebase SDK singletons lazily inside methods/getters that run only after app bootstrap or explicit test setup, or shell/widget tests will fail with `core/no-app`.
- **Habits SWR Cache Must Persist Active Subsurfaces**: In `apps/mobile`, stale-while-revalidate habits UX only stays warm if the cache payload includes the selected tracker detail, activity feed entries, and the newer tracker use-case/composer metadata in addition to the tracker list. Caching only the list makes Today look warm while Activity/detail still cold-start and can regress specialized measurement/workout cards after restore.
- **Android Google Sign-In `canceled` Can Mean Misconfiguration**: In `apps/mobile`, do not assume `GoogleSignInExceptionCode.canceled` from `google_sign_in` on Android means the user backed out. With Credential Manager, that code can also be returned after account selection when SHA/package/server client configuration is wrong. Do not silently swallow it; surface an error or fall back to browser OAuth, and verify flavor-specific Android package IDs plus signing SHAs when native sign-in works on iOS but not Android.
- **Android Splash Asset Safe Area**: For `apps/mobile` splash logos generated through `flutter_native_splash`, do not reuse edge-to-edge transparent logo PNGs directly on Android. Prefer branded light/dark splash images whose artwork is already inset from the canvas edges, and wire them through `image_android` / `image_dark_android` plus dedicated `android_12.image` assets with extra inset, so Android 12's masked splash icon does not visibly clip the outer strokes even when iOS renders the same artwork correctly.
- **Flutter Apple CI Must Not Force CocoaPods Repo Updates**: In `apps/mobile` GitHub Actions for iOS/macOS, do not add unconditional `pod repo update` steps before `flutter build`. Flutter/CocoaPods already resolves pods during the build, and forcing a repo update increases CDN and `raw.githubusercontent.com` traffic enough to trigger `429 Too Many Requests` failures for pods such as `FirebaseCoreExtension` on clean runners.
- **Mobile CI Should Avoid `very_good test` For This Repo**: In the `apps/mobile` GitHub Actions workflow, do not route the full suite through `very_good_workflows/flutter_package.yml` or `very_good test -j ... --optimization --coverage --show-uncovered`. That combination has produced parallel widget-test timeouts, duplicate failure reporting, and runner hangs with stale `Run Tests` logs on both PR and `main`. Run `flutter test` directly with explicit timeouts and deterministic concurrency instead.
- **Flutter AnimatedSwitcher Tests Must Assert Settled Target State**: For `apps/mobile` widget tests around multi-step auth/forms that use `AnimatedSwitcher` or `AnimatedSize`, do not assert that outgoing controls disappear immediately after a tap. Ensure the trigger is visible before tapping, then assert the destination state (for example back button, field contents, focused step) after the transition settles; outgoing children can remain mounted briefly during the animation stack.
- **Dashboard Overlay UX**: For compact+expandable dashboard widgets near chat actions, avoid icon-only rails; use labeled compact triggers and reserve layout space to prevent overlap at desktop breakpoints.
- **Web Auth Shell Consistency**: When changing login/signup/reset-password flows in `apps/web`, preserve the established auth-page shell (background treatment, centered brand block, and overall card language) unless the user explicitly asks for a full auth-screen redesign. Prefer updating the inner form flow first and keep new steps visually subordinate to the existing auth chrome.
- **Web Auth Identify-Step Gating**: On the first `apps/web` login step, do not expose competing password-vs-OTP primary actions before the OTP enablement/settings query resolves. Keep a single primary email CTA, disable it while auth mode is still loading, and reveal OTP entry or password fallback only after the submit path is known.
- **Web OTP Rate-Limit UX**: In `apps/web` login flows, an OTP send response with a retry window is not a dead-end. Advance to the OTP step anyway, preserve a password fallback, and surface the resend cooldown inline so users who already received a code can still complete sign-in.
- **Satellite Cross-App Auth Requires Two Registrations**: When onboarding a satellite app into centralized web login, update both `packages/utils/src/internal-domains.ts` and the satellite's `/api/auth/verify-app-token` route in the same patch. Missing the domain map makes `returnUrl` look invalid on web; missing the route breaks `/verify-token` after redirect.
- **Web Auth Flow E2E Parity**: When the `apps/web` auth UI changes between single-step and multi-step login, update Playwright helpers/specs in the same patch (especially `apps/web/e2e/helpers/auth.ts` and unauthenticated auth specs). Do not assume query params like `?passwordless=false` still make the password field visible on first render.
- **Mobile Auth OTP Availability Must Degrade Gracefully**: In `apps/mobile` login flows, OTP enablement may depend on the app-version bootstrap state, but the identify-step CTA must never spin forever waiting for it. Use a bounded grace window, then allow password fallback while the version check continues in the background.
- **Mobile Auth Errors Must Stay Stage-Scoped**: In `apps/mobile` multi-step login flows, transient errors from one branch (for example OTP send cooldowns) must be cleared or suppressed when the user switches to another branch such as password login. Do not let stale auth errors leak across stages.
- **Persistent FAB Content Insets**: When introducing a persistent floating action button across tabs/routes in a surface, add matching bottom list/scroll padding (for example ~96 on mobile) in every tab/empty/error state so content and controls are never obscured by the FAB.
- **Workspace Selector Special-Case Visibility**: If a workspace has product-wide semantics (for example `ROOT_WORKSPACE_ID`), give it a first-class labeled category in every workspace-picker surface and keep Home/dashboard chrome bound to `currentWorkspace`, not a hidden fallback workspace, so switching context changes both the title and the data.
- **External Project Studio Needs A Generic Fallback Editor**: Do not leave non-`yoola` adapters on a placeholder-only studio screen after import. If imported collections and entries exist, every adapter must at least expose a generic collection browser, entry editor, raw profile/metadata JSON editing, and basic asset management so root-bound content remains visible and operable before bespoke adapter UI lands.
- **Assistant Workspace Binding Must Follow Current Workspace**: On mobile assistant/dashboard surfaces, do not bind workspace-scoped credits, models, or composer controls to `personalWorkspaceOrCurrent`. Use `currentWorkspace` first and only fall back to personal when no active workspace is selected, or team workspaces will silently inherit personal-only assistant behavior.
- **Chat Composer Auto-Hide Must Use Threshold Anchors, Not Build-Time Gating**: In mobile chat surfaces with docked composers, do not tie FAB/collapsed state to transcript presence or generic scroll direction assumptions. Hide the composer only after the user scrolls past a resettable distance threshold, and reset that anchor whenever the composer is explicitly restored so tapping the FAB does not immediately collapse again at the same offset.
- **Assistant Conversation Switches Must Reset To FAB Mode**: On mobile assistant surfaces, treat a conversation change (`chat.id` / restored chat / new conversation) as a fresh context and explicitly collapse the composer back to FAB mode. Do not let the next chat inherit the previous conversation’s expanded composer state, or history switching will feel inconsistent and cover transcript content unexpectedly.
- **Assistant Transcript Messages Must Render Markdown, Not Plain Text**: In mobile assistant transcript bubbles, do not render assistant or user message bodies with raw `Text(...)`. Use the shared markdown renderer (`MarkdownBody` with app theme/link handling) for both primary content and reasoning/transcript blocks so lists, code fences, links, emphasis, and other markdown formatting render correctly in every message.
- **Assistant Mermaid Markdown Needs An Offline Renderer And Fullscreen Viewer**: On mobile assistant transcript surfaces, Mermaid support is not just a style change. Parse ```mermaid``` fences through a custom markdown block syntax, render them with a bundled local script asset inside the app, and provide a dedicated fullscreen viewer with zoom support. Do not depend on CDN-hosted Mermaid scripts or leave diagrams trapped in non-zoomable inline previews.
- **Assistant Mermaid Zoom Controls Must Not Rely On Native WebView Zoom APIs**: In the mobile assistant Mermaid viewer, do not call platform `zoomIn` / `zoomOut` / `zoomBy` helpers directly. - **Special Tag Prompt Rules**: Custom tags like `@<FOLLOWUP>` must not contain internal whitespace, but blank lines between distinct prompt sections are required.
- **Module Boundaries**: When file grows beyond roughly 500 LOC, split it by concern and keep the original entrypoint as a thin barrel re-export so dispatcher imports remain stable.
- **Board Workspace ID Fallback Propagation**: In shared board surfaces, normalize `effectiveWorkspaceId = board.ws_id ?? workspace.id` once near the top-level view and pass that explicit id through headers, dialogs, filters, forms, and mutations. Do not let child actions keep reading raw `board.ws_id`, or some board payloads will still fail list/board mutations with local `"Workspace ID is required"` errors even though the page already knows the correct workspace.
- **Cache-Driven Task List Rollbacks**: When a board view renders columns/lists from the TanStack `['task_lists', boardId]` cache, list rename/delete mutations must restore cached `task_lists`/`tasks` snapshots on error. Do not rely on `invalidateQueries` alone for rollback, because cache-driven list UIs may not refetch immediately and will otherwise stay stuck in the optimistic state.

### 6.3 Security & Validation

- **Proxy Guard Parity**: When adding proxy-level request protections (for example payload caps, rate limits, or body validation), apply them across every app `src/proxy.ts` that fronts API traffic, not just `apps/web`. Shared guard utilities are preferred so rate-limit thresholds, trusted-source bypasses, and rejection payloads do not drift between apps.
- **Hex Color Validation**: When a label or theme input accepts arbitrary color values, validate the submitted value directly with a hex-color regex instead of constraining it to a preset palette enum unless the UX explicitly requires presets.
- **Cloudflare/Vercel Client IP Precedence**: In abuse protection or rate limiting paths that run behind Cloudflare -> Vercel -> Supabase, prefer `cf-connecting-ip` first, then `true-client-ip`, before falling back to `x-forwarded-for` or `x-real-ip`. Generic forwarded chains can otherwise key counters and audit rows on proxy hops instead of the real caller.
- **Authenticated Storage API Pattern**: For server routes that gate Supabase Storage access with session-based auth or workspace membership checks, perform the authorization check with the user-scoped client first, then run the actual `storage.from(...).list/createSignedUrl/createSignedUrls/move/remove` call through `createDynamicAdminClient()`. Mixing user-scoped storage calls into these routes can fail RLS unexpectedly and drift from the repo’s hardened storage API pattern.
- **Rotating Storage Redirects Must Be Non-Cacheable**: When an app route provides a stable/export URL by redirecting to a freshly minted storage signed URL, set `Cache-Control: no-store` on the redirect response. Cached 30x responses can pin an expired provider URL and make “stable” public asset links break unexpectedly.
- **Drive Export Tokens Need A Dedicated Signing Secret**: Do not reuse `SUPABASE_SECRET_KEY`, cron secrets, or other platform-wide credentials to sign workspace Drive export/public-link tokens. Use a dedicated env var (for example `DRIVE_EXPORT_SIGNING_SECRET`) so export-link rotation does not couple unrelated security domains.
- **Rotating Export Tokens Must Pin The Backing Provider**: If a Drive export/public-link token is minted after listing a concrete storage backend, include that backend in the signed token payload and honor it when later resolving signed read URLs. Do not re-resolve against the workspace’s current provider at request time, or provider flips can silently retarget or break previously issued stable links.
- **Export Origins Must Fail Closed Outside Dev**: For share/export URL generation, do not fall back to a hard-coded production hostname in non-dev environments. Require an explicit app origin env (`WEB_APP_URL`/`NEXT_PUBLIC_WEB_APP_URL`/`NEXT_PUBLIC_APP_URL`) and throw when missing so preview and self-hosted deployments do not leak tokenized paths to the wrong host.
- **Archive Download Limits Must Be Enforced While Streaming**: For ZIP or other user-supplied archive downloads, do not rely on `response.arrayBuffer()` and then validate the final buffer size. Read the response stream incrementally, reject once the byte cap is exceeded, and cancel/drain the reader before aborting so oversized archives cannot fully buffer in memory.
- **Self-Hosted Proxy Workers Must Fail Fast On Missing Required Secrets**: For helper services such as unzip proxies, validate required shared-secret env vars at startup instead of only during request handling or health checks. A container that can never authenticate real traffic must not report healthy.
- **Proxy Outbound Fetches Need Bounded Timeouts And Status Mapping**: In self-hosted proxy workers, wrap every outbound `fetch` with a shared abort timeout and map caller-driven validation failures to 4xx/413 plus upstream timeout/download failures to 5xx/504/502. Do not leave proxy jobs hanging indefinitely or collapse permanent ZIP/input errors into generic 500 retries.
- **Proxy URL Payloads Must Be Parsed Before Fetching**: In self-hosted proxy request handlers, validate callback/source URLs with `new URL(...)` and enforce `http`/`https` schemes before any outbound fetch starts. Invalid or unsupported URL payloads are caller errors and must return 400 instead of leaking through as transport failures.
- **Placeholder-Heavy Storage Exports Need A Second Pass Before Enforcing File Count Limits**: When Drive export flows list raw storage objects that include folder placeholder blobs, do not trust a raw `limit + 1` result after filtering placeholders. If the raw scan hit its limit, rerun or otherwise prove the exportable-file count before allowing a “<= N files” export to proceed.
- **Mira Internal Data Client**: Once a Mira chat request has already verified the caller and workspace access at the route boundary, pass an admin Supabase client into Mira context builders and tool executors, and keep every query scoped with explicit `user_id` / `ws_id` predicates. Session-scoped clients can still hit table-permission/RLS failures for internal tool queries even though the route itself has already authorized the action.
- **Schema-Verified Mira Scoping**: When hardening Mira tools that now run through `sbAdmin`, verify the local Supabase schema before adding workspace filters. Do not assume every table has `ws_id` or `user_id` directly; for example `tasks` is workspace-scoped via `board_id`/`list_id`, and some bridge tables inherit scope only through foreign keys. Use the local Supabase MCP endpoint at `http://127.0.0.1:8001/mcp` (or the local Postgres schema) to confirm columns and relationships before patching tool predicates or tests.
- **Mira Prompt Context Permission Parity**: If Mira prompt/context building runs with an admin Supabase client, gate every permissioned context section with the same workspace permission checks used for tool execution, and fail closed if permission resolution is unavailable. Otherwise the prompt can leak calendar/finance/workspace data even when the corresponding tools are blocked.
- **System Route Proxy Exemptions**: API webhook routes and platform-managed cron endpoints (for example Vercel cron targets) that authenticate with platform-wide/shared credentials must bypass proxy IP rate limiting only after the proxy confirms the expected secret/token/signature headers are present. Do not exempt these routes by pathname alone.
- **Persisted Webhook Retries Must Reopen Only Failed Rows**: For webhook-event ledgers such as SePay, treat `received` rows as in-flight duplicates, only reopen rows marked `failed`, and persist any already-created child record id on failure before returning a retryable response. Otherwise duplicate deliveries can either get stuck behind a stale pre-check or create duplicate transactions after partial success.
- **Chat Attachment Type Parity**: When adding a new attachment type for Mira chat, update all three surfaces together: `chat-input-bar.tsx` accept list, `/api/ai/chat/upload-url` extension allowlist, and `/api/ai/chat/file-urls` extension→MIME mapping.
- **Office/Binary Attachment Handling**: For chat uploads that may be rejected by storage MIME checks (for example `.xlsx`/`.docx`), use a safe upload MIME fallback (`application/octet-stream`) and retry once without explicit `Content-Type`. In AI provider routes, never pass unsupported binary MIME types as inline `file` parts; convert them into explicit text notices instead.
- **Mira Workspace Context Sync**: When a Mira tool changes workspace context, update both the mutable server-side `MiraToolContext` used for later tool calls in the same response and the client-persisted chat config used for future turns; otherwise current-turn tools and later user requests will drift to different workspaces.
- **Fixed-Cost AI Reservations**: For fixed-price AI operations that call external services (for example MarkItDown conversion), reserve credits atomically before the external call, then commit the reservation on success or release it on every failure path. Do not rely on a soft allowance pre-check plus post-hoc deduction.
- **Experimental AI Tool Gating**: When an AI tool is highly experimental or prone to recursive failure loops in production, conditionally omit it from both the stream tool definitions and the dynamic system prompt directory when `!DEV_MODE`. Do not rely solely on system prompt instructions to stop models from calling unstable tools.
- **AI SDK Tool Part Parsing**: In Mira chat UI code, do not assume tool parts expose `.toolName` or `.toolCallId`. Resolve tool identity with the AI SDK helper first and fall back to the `type`/top-level `toolCallId` fields so visual tools (for example `render_ui`) and context-switch side effects still work with serialized message parts.
- **Mira Retry After Stop**: In Mira chat queueing, do not call `sendMessage` in the same tick as `stop()` while `useChat` is still `submitted`/`streaming`. Queue the retry and flush only after the hook reports an idle status, or follow-up user turns can append locally while the assistant remains stuck in `Thinking...`.

### 6.4 Tooling & CI

- **Targeted Test Runs**: For single-package/unit validation, run the package-local test runner directly (for example `bun --cwd packages/ai vitest run src/tools/executors/timer.test.ts`) instead of `bun test <path>` from repo root, which fans out to monorepo-wide `turbo run test`.
- **`apps/web` Route Tests Use Bun's Limited `vi` Shim**: When adding `apps/web` route tests that run via `bun test`, do not rely on Vitest-only helpers like `vi.hoisted`, `vi.importActual`, or `vi.resetModules`. Prefer plain object mocks and explicit stub exports for any transitive imports pulled in by the route module.
- **Global Check Baseline Drift**: If `bun check` fails solely because repo-wide tool versions drift (for example Biome schema/CLI mismatch), do not modify unrelated workspace configs in feature PRs; complete scoped verification (for example `bun check:mobile` for mobile-only work) and report the pre-existing global failure explicitly.
- **Scoped Biome Writes**: If `bun ff` or repo-wide `biome --write` hits permission errors in non-worktree support directories (for example `.agents/**`), rerun Biome only on the touched workspace files instead of forcing broader filesystem changes. Keep formatting and import-order fixes scoped to the files you actually modified.
- **Large Generated Files Need `files.maxSize` In Biome**: If a checked-in generated artifact such as `packages/types/src/supabase.ts` grows past Biome's default 1 MiB processing limit, raise `biome.json` `files.maxSize` instead of relying on narrow ignore patterns. Repo-wide `bun ff` passes can still surface the warning and fail CI unless the project-wide size limit matches the generated file reality.
- **Dart Formatting via Mobile Check**: Do not run `bun ff` with Dart-only paths (for example `apps/mobile/**/*.dart`); Biome ignores Dart files and can fail with "No files were processed." Use `bun check:mobile` (or `dart format` in `apps/mobile`) for Flutter formatting and verification.
- **Broken Bun CLI Binaries**: If a `bun run` script reports a repo-local CLI as missing but the package is installed, inspect the resolved `node_modules/.bun/*` package payload and the `.bin` symlink target before changing PATH or shell setup. After any CLI dependency bump, verify the installed package actually contains the declared bin target, not just the manifest entry. If the published version is missing its built CLI files, prefer pinning to a known-good package version with `bun add`; only add a repo-local compatibility wrapper when the requested version must stay in place, and make the wrapper prefer the upstream CLI automatically once the package is fixed.
- **Local-Only E2E Admin Resets**: Any E2E helper that uses Supabase service-role credentials to reset local state (for example rate-limit counters) must refuse non-local Supabase URLs and should default only to the known local stack values (`http://127.0.0.1:8001` and the standard local secret key). Never let Playwright or test helpers derive destructive reset targets from arbitrary `.env` files or remote cloud URLs.
- **E2E Auth Setup Must Validate A Server-Readable Session**: When Playwright setup saves authenticated `storageState`, do not treat a client-side authenticated fetch alone as sufficient proof of login. After any helper login flow, force at least one server-rendered protected navigation (or equivalent cookie-backed check) before writing `storageState`, or setup can pass while later page navigations and `context.request` calls still get redirected to `/login` / `401`.
- **Dev E2E Login Must Set Cookies Server-Side**: For local/dev Playwright auth bootstraps, do not treat raw Supabase magic links or URL hash fragments as a complete login path. Use a dev-only server route or equivalent request-bound flow that verifies the token and writes the real auth cookies before setup saves `storageState`.
- **Upstash Ratelimit Redis Adapters**: If you wrap or narrow the Redis client passed to `@upstash/ratelimit`, expose both `evalsha` and `eval` in addition to `get`/`set`. The package falls back to `eval(...)` on `NOSCRIPT` even though its constructor types may appear to require only `evalsha`.
- **Incremental Type-Check Drift**: If CI fails a package `type-check` while the same `tsgo --build` command passes locally, rerun the package-local build with `tsgo --build --force` (or otherwise invalidate the package `.tsbuildinfo`) before blaming GitHub Actions caching. Preserve normal caching for day-to-day runs, but route `tsgo --build` scripts through the shared cache-invalidation wrapper so `bun.lock`, root runtime pin, or package config changes clear stale `.tsbuildinfo` automatically.
- **Workflow Bun Parity**: Keep the Type Check workflow Bun version aligned with the repo `packageManager` pin so local `bun run type-check` and GitHub Actions resolve the same Bun runtime behavior.
- **Flutter iOS CI Dependency Cache Scope**: In `apps/mobile` GitHub Actions, cache CocoaPods state (`ios/Pods`, CocoaPods caches/spec repos) and Flutter dependency artifacts with keys derived from `pubspec.lock`, `Podfile`, and `Podfile.lock`, not the entire app tree. Do not skip `flutter pub get`, simulator boot, or the actual iOS build on cache hits; broad cache keys and build-skipping gates turn routine app diffs into fresh CocoaPods CDN fetches and can trigger `raw.githubusercontent.com` `429` failures.
- **Shell Path Quoting for Route Segments**: When running git or shell commands against files whose paths contain route-group characters such as `(`, `)`, `[`, or `]`, always quote the full path argument (for example `"apps/web/src/app/[locale]/(dashboard)/..."`) to avoid shell parsing errors.
- **Root Script Test Coverage**: When adding or refactoring root-level `scripts/*.js` utilities that are not covered by workspace package tests, add a dedicated repo-root test entry (for example `node --test scripts/foo.test.js`) and wire it into `scripts/check.js` so regressions are caught by `bun check`.
- **Check Runner Flag Semantics**: Keep `scripts/check.js` fail-fast by default; use an explicit opt-out flag (currently `--run-all`) when full check execution is needed for diagnostics.
- **Unrelated `bun check` Failures Need Scoped Follow-Up**: If repo-wide `bun check` fails in untouched packages, do not patch unrelated domains just to get green. Run focused verification for touched files/workspaces (for example package-local tests or `bun check:mobile`) and report the unrelated failing package/test clearly in the handoff.
- **Workspace Subpath Imports**: Before importing a workspace package subpath like `@tuturuuu/pkg/foo`, verify that the target is actually covered by that package’s `exports`. Do not assume a directory `index.ts` is reachable via the bare directory subpath; prefer an explicitly exported file path or a relative import within the same package.
- **Server-Side Internal API Auth Forwarding**: In Server Components or other server-only loaders that call `@tuturuuu/internal-api`, always forward incoming `cookie`/`authorization` headers via `withForwardedInternalApiAuth(headers())`. Without forwarded auth headers, internal API calls can return unexpected `401` even when the page itself already has an authenticated user.
- **Shared List Hook/API Page-Size Parity**: When a shared client hook hardcodes a list page size (for example group pickers in settings), keep the backing API validator cap at or above that same value and prefer admin-backed reads after permission checks for settings/report selectors that must see the full authorized list. A low validator cap or user-scoped read path can fail silently as a `400` or return an incomplete option set.
- **Bearer-Aware API Clients**: For API routes that must serve web sessions and mobile Bearer tokens, initialize Supabase with `createClient(request)` in the route handler instead of cookie-only helpers/wrappers so Authorization headers are honored across clients.
- **Auth Before Workspace Normalization**: In request handlers that call `normalizeWorkspaceId(...)`, authenticate (`supabase.auth.getUser()`) before normalization when possible. This avoids surfacing normalization/auth lookup failures as generic `500` responses and keeps unauthenticated requests on explicit `401` paths.
- **Proxy-Only Table Ownership Enforcement**: When a route reads or mutates a `public` table that is listed in `packages/supabase/src/next/protected-tables.ts`, assume `createClient(request)` will delegate that table to an admin proxy client and therefore bypass RLS. Re-apply the intended ownership or membership predicates explicitly inside the API route before returning or mutating rows.
- **Protected Finance Table CRUD Split**: In finance API routes that already perform explicit workspace permission checks, use `createAdminClient()` for `wallet_transactions` / `wallet_transaction_tags` verification and mutations, and reserve `createClient(request)` for caller auth plus user-scoped RPCs/redacted reads. Request-scoped direct CRUD on those protected tables can fail with table-permission errors even after permission checks pass.
- **Proxy-Only Rollout Gate**: Do not add a table to `packages/supabase/src/next/protected-tables.ts` until the app-facing server components, client hooks, and shared helpers for that table have already been migrated off direct Supabase CRUD. If a legacy domain still relies on direct table access in shared UI or utilities, keep the table out of the proxy-only list until that migration is complete.
- **Permissioned List API Scope Enforcement**: For workspace list endpoints that read through `createAdminClient` and support optional `userId` filters (for example time-tracking requests), always resolve caller permissions first. If the caller lacks the relevant manage permission, force `user_id = auth user` on both count and data queries and reject explicit filters targeting other users.
- **Non-Permission Workspace API Gates Must Enforce MEMBER Type**: For workspace-scoped routes that are not protected by `getPermissions`, do not rely on `workspace_members` row existence alone. Use `verifyWorkspaceMembershipType(...)` from `@tuturuuu/utils/workspace-helper` (default `requiredType: 'MEMBER'`) or an equivalent `.eq('type', 'MEMBER')` check so `GUEST` callers are denied by default.

### 6.5 Type Safety & Platform Details

- **API Route UUID Params**: In API routes, validate UUID path params with shared `zod` schemas (`z.guid()` + `safeParse`) instead of ad-hoc null/regex checks.
- **GUID-Validated Test Fixtures Must Use Real GUIDs**: When API route schemas validate request body IDs with `z.guid()` (for example `tag_ids`, wallet IDs, category IDs), keep Vitest fixtures on real GUID-shaped values. Placeholder strings like `tag-1` belong only in read-side mocked payloads that are not schema-validated.
- **Zod JSON Payload Typing**: When a request field is stored as Supabase `Json` (e.g. `snapshot`), avoid leaving it as `unknown` after `zod` parsing. Use a `zod` transform (e.g. `z.unknown().transform((v) => v as Json)`) so the parsed payload matches typed Supabase `.update(...)` inputs and downstream helpers.
- **Supabase Helper Extraction**: When moving Supabase DB writes into shared helper modules, prefer structural interfaces (or thin generic adapters) over concrete `createAdminClient` return types to avoid cross-package generic incompatibilities during type-check.
- **Supabase Client Typing Alias**: For Supabase client type annotations in route/helpers, use the canonical `TypedSupabaseClient` alias instead of `Awaited<ReturnType<typeof createAdminClient>>` so typing stays stable across package boundaries and helper signatures.
- **Supabase Pagination Helper Typing**: When extracting reusable paginated Supabase readers, type the page callback as returning an awaitable/thenable query result (`PromiseLike<{ data, error }>` or equivalent), not a concrete `Promise`. PostgREST filter builders are awaitable but do not satisfy the full `Promise` interface, and over-constraining the helper will pass runtime tests while failing repo type-check.
- **Internal API Barrel Type Parity**: When adding a new response/interface export to `packages/internal-api/src/*` that UI code will import through `@tuturuuu/internal-api`, update the package barrel in `packages/internal-api/src/index.ts` in the same patch. Exporting only the runtime helper function will pass local call sites until consuming apps type-check against the barrel.
- **RPC Return Enums Must Cast To Declared Scalar Types**: When a `RETURNS TABLE` RPC exposes enum-backed columns such as `platform_icon` through `text` fields for API compatibility, cast those columns explicitly inside `RETURN QUERY` (`icon::text`). PostgreSQL will reject the function at execution time even when the enum’s labels are textual.
- **Signed Upload Finalization Must Not Reclassify A Successful PUT As Failure**: When a client helper performs a signed file upload and then calls a follow-up finalize endpoint, do not throw away the successful upload result if finalize fails. Return the upload success plus finalize status separately so callers do not retry and create duplicate blobs.
- **External Provider Fetches Need Explicit Timeouts**: For request-path integrations that call third-party APIs (OAuth exchange, token refresh, webhook provisioning, account sync), always wrap `fetch` with a bounded timeout such as `AbortSignal.timeout(...)` and translate timeout failures into clear upstream/provider errors. Do not leave Next.js route handlers waiting on unbounded external I/O.
- **Cursor-Based Third-Party Syncs Must Retry 429s Before Failing**: When a provider list endpoint requires pagination (`since_id`, cursors, next-page tokens), do not stop at the first rate-limit response. Honor `Retry-After` when present, retry a bounded number of times, and only fail the sync after those retries are exhausted.
- **Provider-Specific Rate-Limit Headers Must Be Honored Before Generic Fallbacks**: If a third-party API documents a custom retry header (for example SePay's `x-sepay-userapi-retry-after`), prefer that header over generic `Retry-After` parsing in pagination and sync loops.
- **Auto-Resolved External Wallet Links Must Fail Closed On Ambiguity**: For bank-integration webhook routing, do not auto-create a new wallet or silently pick a candidate when multiple active links match the same payload. Log the ambiguity and fail closed so operators can relink explicitly instead of misrouting transactions.
- **Webhook Amount Normalization Must Preserve The Provider Contract**: If a provider already supplies transaction direction separately (for example SePay `transferType`), normalize incoming amount fields to positive magnitude and derive the signed ledger amount from the direction field later. Do not let local validation changes reinterpret valid debit payloads as negative normalized amounts unless the provider contract explicitly changed.
- **Finalize Status Requires Caller Follow-Through**: If a signed-upload helper returns `{ finalize: { success, error } }` instead of throwing, every caller must explicitly consume that status. Callers that require the follow-up work must throw on `success: false`; callers with optional follow-up work must surface the warning to the user instead of silently treating the upload as fully complete.
- **Central Storage Path Helpers Must Sanitize Traversal Segments**: Shared workspace-storage path builders (for example `normalizeRelativePath`, `buildWorkspaceStorageKey`, and prefix helpers) must run all relative inputs through `sanitizePath` or equivalent traversal-safe validation. Do not rely only on API route callers to strip `..` segments before building backend object keys.
- **Storage Terminal Names Need Their Own Sanitization**: Do not sanitize only parent paths when building workspace storage keys. Filenames, folder names, and rename targets (`filename`, `folderName`, `currentName`, `newName`) must be sanitized as terminal segments too before any `posix.join(...)`, or traversal and nested-path injection can re-enter through the final path component.
- **Workspace Storage Quota Checks Must Be Overwrite-Aware**: Capacity enforcement for Drive uploads must subtract the existing object size when `upsert`/overwrite is allowed, and duplicate-name `409` checks must run before quota checks on non-overwrite paths. Do not treat every upload as net-new bytes or replacements near the limit will fail incorrectly.
- **JSON Route Bodies Need Parse Guards Before Schema Validation**: In Next.js route handlers, never pass `await request.json()` directly into `safeParse(...)` when malformed JSON should return 400. Parse inside a `try/catch` first, then validate the parsed value with Zod so invalid JSON does not fall through to the generic 500 handler.
- **Typegen vs Published Dist Drift**: After `bun sb:typegen`, do not assume workspace consumers of `@tuturuuu/types` immediately see new RPC names through their exported package typings. If the checked-in `dist/` declarations still lag, keep the fix local with a narrow RPC wrapper/cast at the new call site rather than widening shared client types or forcing an unrelated package build.
- **Supabase Update Payloads Must Use Concrete Table Types**: After schema/typegen changes, do not pass `Record<string, unknown>` / `Record<string, any>` into `.update(...)` or `.insert(...)` on typed Supabase builders. Use `TablesUpdate<'table'>` / `TablesInsert<'table'>` (or `Database['public']['Tables'][...]['Update']`) and coerce nullable booleans/union values explicitly, or repo type-check will fail once reject-excess-property typing tightens.
- **Queue Send Paths Must Pre-Skip Blacklisted Emails**: Notification and post-email queue processors must consult the infrastructure blacklist before attempting delivery, preferably through the existing batch RPC (`get_email_block_statuses`) with a per-run cache/prefetch. Do not rely on downstream email-service blocked-recipient responses as the first blacklist signal, or queues will waste send work and surface inconsistent skip timing.
- **Type-Safe Group Iteration**: In strict TS files with discriminated unions and `noUncheckedIndexedAccess`, avoid `array[index]` iteration for render groups. Prefer `for (const [i, item] of array.entries())` plus `switch (item.kind)` to preserve narrowing and prevent `"possibly undefined"` regressions.
- **Broken Package Export Types Under Bundler Resolution**: If a dependency runs correctly but `moduleResolution: "Bundler"` cannot resolve its exported typings (for example `isomorphic-dompurify`), do not rely on ambient shim declarations in import-bearing `global.d.ts` files or patch the dependency ad hoc. Prefer direct browser-only imports in client components and typed runtime loaders (`await import(moduleName)` with a string variable plus a local interface cast) in server/shared code so runtime behavior stays intact while type-check remains stable.
- **Shared JSONContent Types**: Do not import `@tiptap/react` (or other UI-only packages) into `@tuturuuu/types` just to re-export `JSONContent`. Define a local `JSONContent` shape that is compatible with the repo’s `Json` type (attrs/marks attrs should be `{ [key: string]: Json | undefined }`) so it can be stored/typed as `Json` in app code.
- **Workspace Runtime Export Drift**: When adding new runtime exports (schemas/constants/functions) to a shared workspace package, do not assume sibling packages that import the published package entrypoint will see those symbols immediately in local tests. If the consumer resolves the built package output instead of source, either update the consumer to provide a local wrapper for the new runtime exports or explicitly verify the workspace dependency path before relying on the new symbol.
- **External App EPM Integrations Should Start With One Server Adapter**: For sibling/public apps that consume EPM delivery (for example Yoola), replace local hardcoded content registries with one server-side adapter module that fetches the delivery payload once, maps it into app-specific view models, and passes props into client components. Do not spread raw SDK calls or duplicate fallback content logic across multiple route files.
- **External EPM Delivery Apps Should Hydrate One Shared Query**: If a branded/public app consumes EPM delivery with TanStack Query, prefetch and hydrate the delivery query once at the app shell/layout boundary, keep adapter normalization centralized, and let route components read from that shared cache. Do not let each route refetch the same delivery payload or duplicate normalization logic just to support client-side navigation or infinite scroll.
- **Featured Rails Should Derive From Delivery Content, Not A Shadow Contract**: When an external site or EPM preview needs a featured carousel/rail, order those items from existing delivery entries plus section/profile `profile_data` hints such as featured slugs. Do not introduce a second frontend-only featured payload that can drift from the delivered content.
- **Featured Rails Should Not Fallback When Config Is Empty**: If a featured carousel/rail is driven by section/profile featured slugs, hide that special surface when no featured slugs are configured. Do not silently fall back to the first published media or entry, or operators cannot tell whether the featured config is actually working.
- **Singleton Section Featured Controls Should Be Edited In EPM, Not Raw JSON**: For integrations like Yoola that use `singleton-sections` entries as page configuration, expose structured featured-entry controls on the relevant section entry (`gallery`, `writing`) and persist canonical slug arrays such as `featuredArtworkSlugs` or `featuredEntrySlugs`. Do not require operators to hand-edit `profile_data` JSON for routine featured ordering.
- **Featured Ordering Editors Should Use One Compact Ordered List**: When EPM lets operators select and order featured entries, do not duplicate the same items in a separate "selected" list plus another "available" checklist. Keep one compact list where selection state and up/down ordering controls live together.
- **Candidate Entry Editors Should Offer Direct Featured Toggles Without Forking State**: If an artwork or writing entry can be featured externally, expose add/remove/reorder controls directly in that fullscreen entry editor too, but make those controls mutate the same singleton section query/cache state rather than storing a second per-entry featured flag.
- **Yoola Taxonomy Should Be Managed From Structured EPM Fields**: When external-project delivery depends on entry taxonomy such as artwork categories or writing tags, expose dedicated `category` and `tags` controls in the EPM entry editor and persist them to `profile_data.category` / `profile_data.tags`. Do not send operators to raw metadata/profile JSON for routine taxonomy edits.
- **Yoola Taxonomy Suggestions Must Come From Section Config, Not Corpus Inference**: For `gallery` / `writing` style singleton-section integrations, treat `profile_data.categoryOptions` and `profile_data.tagOptions` on the relevant section entry as the canonical picker/filter registry. Do not populate “existing categories/tags” by scanning every entry in the collection, or operators will see stale phantom taxonomy that is not actually configured for delivery.
- **Large EPM Entry Editors Must Split Query Logic From Presentational Modules**: When the fullscreen EPM entry editor grows, keep TanStack Query state, mutations, and derived workspace data in `entry-detail-client.tsx`, and extract sticky headers, taxonomy cards, featured controls, and sidebar/status panels into adjacent components. Do not let the entry detail route drift back into one 3k-line render file.
